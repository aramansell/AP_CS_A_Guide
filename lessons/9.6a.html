9.6a — Combatant Hierarchy: Interface + Creatures

RPG Build ~25 min

Your DungeonCrawl has one monster type at a time. But a real RPG has goblins, dragons, skeletons, slimes — each with different stats and behaviors. Inheritance makes this clean: one contract, many implementations.

Open the Combat System (Student RPG Guide). Follow Part 1: the class hierarchy.

Build the Hierarchy

Combatant interface:

  • int attack() — return damage dealt
  • void takeDamage(int amount) — subtract from HP
  • boolean isAlive() — HP > 0
  • String getName()
  • int getHp()

Creature abstract class implements Combatant:

  • Protected fields: name, hp, maxHp, attackPower
  • Constructor takes name, hp, attackPower
  • Implements takeDamage(), isAlive(), getName(), getHp()
  • Does NOT implement attack() — leave it abstract for subclasses

Concrete creatures:

  • Goblin extends Creature — 20 HP, 5 ATK. attack() returns ATK + random 1-3.
  • Dragon extends Creature — 80 HP, 20 ATK. attack() returns random 15-25.
  • Skeleton extends Creature — 30 HP, 8 ATK. 30% chance to dodge: takeDamage() reduced to 0.

Player now implements Combatant. Your existing Player class already has attack(), takeDamage(), isAlive(), getName(). Add implements Combatant and a getHp() method.

Socratic Checkpoint

  1. Why is Creature abstract? What happens if you try new Creature("???", 10, 5)?
  2. Skeleton overrides takeDamage(), not attack(). If it has a 30% dodge chance, where does the random roll happen?
  3. Why does Dragon override attack() but not takeDamage()? What's the rule for which methods to override?

Acceptance — your hierarchy must:

  • Combatant interface with 5 method signatures
  • Creature abstract — can't be instantiated directly
  • Three creatures with distinct attack/takeDamage behavior
  • Player implements Combatant with existing + new methods
