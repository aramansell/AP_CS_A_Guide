3.2a — Boolean Logic: Combining Conditions

Discovery Lab ~25 min

Last time you made simple decisions: if X, do Y. But real decisions are rarely that simple. "Did I hit AND is the enemy still alive?" "Are you a goblin OR a troll?" Multiple conditions, one decision.

The Three Boolean Operators

OperatorNameTrue when...
&&ANDBOTH conditions are true
||ORAT LEAST ONE condition is true
!NOTThe condition is FALSE

Tinker: Truth Table Detective

Create a file called BooleanLab.java. Run this code:

boolean a = true;
boolean b = false;

System.out.println(a && b);
System.out.println(a || b);
System.out.println(!a);
System.out.println(!b);

Now try all four combinations of a and b (TT, TF, FT, FF). Before each run, write down what you predict. Then run and check. Build a complete truth table in your notes.

Socratic Checkpoint — Short-Circuit Detective

Open the Boolean Logic docs. Find the section about short-circuit evaluation. Then answer:

  1. What does Java stop checking an AND at the first false?
  2. What does it stop checking an OR at the first true?
  3. Why does if (enemy != null && enemy.isAlive()) protect you from a crash, but if (enemy.isAlive() && enemy != null) might crash your program?

Build: Combat Readiness Checker

Create CombatCheck.java. Your player has:

  • int hp — hit points
  • boolean hasWeapon — does the player have a weapon?
  • boolean isPoisoned — is the player poisoned?

Write a program that prints "Ready for combat!" if the player has a weapon AND HP > 0. Otherwise, print "Not ready!" Test with different combinations of values.

Now extend it: also print "Warning: fighting while poisoned!" if the player is ready for combat AND is poisoned. Think about where this check goes — inside the "ready" branch?

Build: Loot Eligibility

A treasure chest opens if you have a key OR you know the password (let's say the password is "dragonfire"). Write the code that uses Scanner to ask if the player has a key (yes/no) and what the password is, then prints whether the chest opens. Use ||.

Precedence Trap: ! binds tightest, then &&, then ||. So a && b || c means (a && b) || c, not a && (b || c). When in doubt, use parentheses. They cost nothing and prevent bugs.

Checkpoint — before moving on, you must be able to:

  • Build a truth table for AND, OR, and NOT
  • Combine two or more conditions using && and ||
  • Explain short-circuit evaluation and why order matters
  • Use parentheses to control boolean operator precedence
