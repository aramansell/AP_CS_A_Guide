3.6a — Combat Engine: Attack Methods

RPG — Part 1 ~25 min

You've learned if/else, boolean logic, else-if chains, and comparing objects. Now you build something real: a combat engine. Your character can finally fight.

Open the Combat System Student RPG Guide. This is your primary resource for the next three lessons. It provides acceptance criteria — what your code must do to pass — but not the code itself.

Part 1: playerAttack()

Read Part 1 of the guide carefully. Your job: implement a playerAttack method that follows all the rules. The guide tells you the method signature, the behavior for critical hits/critical misses/standard attacks, and gives you an acceptance test.

Create Combat.java and CombatTester.java. In Combat.java, write the playerAttack method. Use your Dice class for the d20 roll.

Socratic Checkpoint — Before You Code

  1. How many different possible outcomes does this method have? What kind of if-statement structure will you need?
  2. The guide mentions that a natural 20 is an automatic hit regardless of attack bonus vs AC. Where in your if/else chain must the critical hit check go? Why?
  3. Look at the acceptance test in the guide. What values does it pass for attackBonus, targetAC, and damageDie? What's the range of possible return values? How will you know if your code is correct?
  4. Why does the method return an int (the damage) instead of printing it? What does the CALLER do with that return value?

Part 2: monsterAttack()

Now read Part 2 of the guide. Monsters don't get critical hits or misses — just standard attacks. Look at your playerAttack code. How much of the standard attack logic is shared? Could you reuse it?

Tinker: Code Reuse

The guide asks: "How much of the logic from playerAttack can you reuse here?" Before writing monsterAttack, think about it. Do you copy-paste the standard attack block? Or is there a way to avoid duplication?

If you can't figure out a clean way to share code, copy-pasting is okay for now. We'll optimize in a later unit. Either way, get both methods working and passing their tests.

Checkpoint — before moving on, you must be able to:

  • Explain the three-outcome if/else if structure (crit hit, crit miss, standard)
  • Implement a method that takes parameters and returns a computed value
  • Use Dice.java as a reusable component inside another class
  • Verify behavior using the provided acceptance tests
