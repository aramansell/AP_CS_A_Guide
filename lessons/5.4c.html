5.4c — BankAccount: Mutators and Validation

Type: Expand Application. Card was read-only — you could only get values. Now add methods that CHANGE state: deposit, withdraw. And guardrails that prevent bad state.

Expand Application ~25 min

The Goal

Build a BankAccount class where money can move. Deposits increase the balance. Withdrawals decrease it — but only if there's enough money. Overdraft prevention built in.

Created account #1001 for Alice with $500.00
Deposited $200.00. New balance: $700.00
Withdrew $100.00. New balance: $600.00
Withdrew $900.00: INSUFFICIENT FUNDS
Final balance: $600.00

Reference: BankAccount Walkthrough — the full reference.

Build It: BankAccount

Create BankAccount.java with these fields:

  • private int accountNumber
  • private String ownerName
  • private double balance

Start with a constructor and getters (same pattern as Card). Then add these methods:

public void deposit(double amount) {
    // YOUR CODE: add amount to balance
    // Print the new balance
}

public void withdraw(double amount) {
    // YOUR CODE: subtract amount from balance
    // BUT FIRST: check if balance >= amount
    // If not, print "INSUFFICIENT FUNDS" and DO NOT change balance
}

Bug Hunt

What happens if someone calls deposit(-50)? Does the balance go down? Should it? Add a guard: if the amount is negative, print an error and don't change the balance. The same guard should go on withdraw().

Socratic Checkpoint

  1. Why is withdraw() a mutator but getBalance() is an accessor?
  2. Where does the overdraft check go — in the BankAccount class or in the tester? Why does it belong inside the class and not outside?
  3. What would happen if balance were public instead of private? How could that break the overdraft protection?

Extend It (pick one)

  • Add a transferTo(BankAccount other, double amount) method that moves money between two accounts.
  • Add a transaction history: store the last 5 transactions and add a printHistory() method.
  • Add interest: a applyInterest(double rate) method that increases balance by rate%.
