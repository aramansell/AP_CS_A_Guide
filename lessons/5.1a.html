5.1a — Anatomy of a Class: The Blueprint

Discovery Lab ~25 min

In Units 1-4, all your code lived in main(). Every variable, every loop, every if statement — jammed into one method. That works for small programs, but your RPG is growing. You have player stats, monster stats, inventory items, combat rounds. main() is not a filing cabinet.

A class is a blueprint that describes an object — its data (instance variables) and its behavior (methods). You've already used classes: String, Scanner, ArrayList. Now you write your own.

Experiment: Read a Class

Open the Anatomy of a Class docs. Read the annotated Dice class. Copy it into Dice.java and compile it. Then create TestDice.java:

public class TestDice {
    public static void main(String[] args) {
        Dice die = new Dice(6);
        System.out.println(die.roll());   // whoa, it returns different values!
        System.out.println(die.roll());   // that's the point
        System.out.println(die.roll());
    }
}

Answer: Where are the instance variables? Where is the constructor? How many methods does Dice have? What does new Dice(6) actually do?

File Name Trap: The class Dice must be in a file named Dice.java. Case matters. If you name it dice.java or DiceClass.java, it won't compile. Every public class must live in a file with the exact same name.

Socratic Checkpoint

  1. Dice has a sides instance variable. Could you access die.sides from TestDice? Why or why not?
  2. Each time you call die.roll(), you get a different result. Where is the randomness coming from, and why doesn't calling roll() change the value of sides?
  3. If you create Dice d8 = new Dice(8) and Dice d20 = new Dice(20), do they share a sides variable or does each have its own?

Checkpoint — before moving on, you must be able to:

  • Identify instance variables, constructors, and methods in a class
  • Explain the difference between a class (blueprint) and an object (instance)
  • Use new to create an object and call its methods
