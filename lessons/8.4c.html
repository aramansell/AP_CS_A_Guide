8.4c — Tile-Based Movement & Wire It In

RPG Build ~25 min

The Goal

Your DungeonMap has rooms, corridors, and entities. Now make the player move through it. Movement on a grid is a state-tracking problem: you need to remember what was on a tile before the player stood there so you can restore it when they leave.

Add Player State to DungeonMap

Add these fields:

  • int playerRow, playerCol — current position
  • char floorUnderPlayer — what was on this tile before 'P'

Update placePlayer() to save both position and the original tile.

Walkable tiles: '.' (floor), 'T' (treasure), '+' (door). Blocked: '#' (wall), out of bounds. 'M' (monster) triggers combat — move onto it.

Socratic Checkpoint

  1. Why do you need floorUnderPlayer? What happens if you just set the old position to '.' every time?
  2. When the player steps on 'T', you trigger treasure. When they step on 'M', you trigger combat. Where should that logic live — in movePlayer() or in the game loop?

Wire Into DungeonCrawl

Replace the room-number system with DungeonMap:

  • Create a DungeonMap (15×15 for a starter dungeon)
  • Add 3–4 rooms, connect with corridors
  • Place player at first room entrance
  • Place 3–5 monsters and 2–3 treasures
  • Game loop: display map, read direction (N/S/E/W), call movePlayer()
  • On 'M': trigger combat. On 'T': add to inventory.
  • Win condition: all monsters defeated

Socratic Checkpoint

  1. Read the docs section on line-of-sight. Why is checking every tile between two points harder than it looks?
  2. Your movement currently lets the player move through monsters to trigger combat. Some RPGs block movement when a monster is adjacent. How would you change movePlayer() to enforce that?

Checkpoint — before moving on, you must be able to:

  • Implement tile-based movement with state tracking
  • Handle boundary checks and walkable-tile logic
  • Wire a 2D map into an existing game loop
