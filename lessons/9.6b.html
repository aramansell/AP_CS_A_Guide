9.6b — BattleEngine: One Loop, Infinite Monsters

RPG Build ~15 min

The Payoff

You have three different creature types, each with unique combat behavior. But your combat code shouldn't care which one it's fighting. This is polymorphism: write one battle loop that works with ANY Combatant — now and forever.

Open the Combat System (Student RPG Guide). Follow Part 2: BattleEngine.

Build: BattleEngine.java

Create BattleEngine.java. ONE static method:

public static Combatant runBattle(Combatant a, Combatant b) {
    System.out.println("=== COMBAT: " + a.getName() + " vs " + b.getName() + " ===");

    while (a.isAlive() && b.isAlive()) {
        System.out.println(a.getName() + " [" + a.getHp() + "] vs "
                         + b.getName() + " [" + b.getHp() + "]");

        // A attacks B
        int dmg = a.attack();
        System.out.println(a.getName() + " deals " + dmg + " damage!");
        b.takeDamage(dmg);

        // B counterattacks (if still alive)
        if (b.isAlive()) {
            dmg = b.attack();
            System.out.println(b.getName() + " deals " + dmg + " damage!");
            a.takeDamage(dmg);
        }
        System.out.println();  // blank line between rounds
    }

    Combatant winner = a.isAlive() ? a : b;
    System.out.println(winner.getName() + " wins!");
    return winner;
}

This method knows NOTHING about Goblin, Dragon, or Skeleton. It works with ANY two Combatants. Test it:

Combatant goblin = new Goblin();
Combatant dragon = new Dragon();
Combatant winner = BattleEngine.runBattle(goblin, dragon);

Socratic Checkpoint

  1. The engine calls a.attack(). If a is a Skeleton, that does one thing. If a is a Dragon, it does another. How does Java know which one to call? What is the name of this mechanism?
  2. Could you add a new creature type — say, Healer that restores HP on attack — without changing a single line of BattleEngine? Why or why not?
  3. What happens if you pass null for either parameter? Where should you add a null check?

Acceptance — your BattleEngine must:

  • Work with any two Combatant implementations
  • Handle Goblin vs Dragon, Skeleton vs Player, Player vs Player
  • Return the winning Combatant
