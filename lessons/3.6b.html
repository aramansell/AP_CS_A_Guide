3.6b — Combat Engine: The Combat Round

RPG — Part 2 ~25 min

You have attack methods. Now combine them into a full combat round: player attacks, monster takes damage, and — only if it's still alive — the monster counterattacks.

Read Part 3 of the Combat System Student RPG Guide.

Socratic Checkpoint — Before You Code

  1. The method has 8 parameters. Trace through one call: which parameter controls the player's roll bonus? Which controls the monster's defense? Which tracks the monster's HP?
  2. Why does the guide say "the monster should ONLY attack the player if the monster is still alive"? What boolean condition checks this? What happens if you forget this check?
  3. Why does combatRound return monsterHP? If we didn't return it, how would the next round know the monster's current HP?
  4. Look at the acceptance test in the guide. It calls combatRound twice with the same monster. What's the point of the second call? What are we verifying?
Trap: The monster should NOT attack if it's dead! This is where boolean logic from 3.2a is critical. What condition do you write after the player's attack to decide whether the monster gets a counterattack?

Tinker: Print the Story

After you get the basic combat round working, make it more readable. Print a separator line before the round. Print the player's attack result with the monster's name. Print the monster's remaining HP. If the monster counterattacks, print how much damage the player took. If the monster dies, print a dramatic death message.

The output should tell a story — not just numbers.

Checkpoint — before moving on, you must be able to:

  • Orchestrate multiple method calls in sequence
  • Use a conditional to decide whether a counterattack happens
  • Track and return state (HP) across method boundaries
  • Trace the flow of a multi-step combat round
