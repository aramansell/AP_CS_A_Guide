9.5a — Downcasting: Getting Your Original Type Back

Discovery Lab ~25 min

When you store a Dragon in a Creature variable, Java forgets it's a Dragon — it only remembers the Creature parts. But sometimes you need the Dragon-specific stuff back. Downcasting tells the compiler "trust me, this Creature is actually a Dragon." It's powerful, dangerous, and the AP exam tests whether you know when it's safe.

Experiment: When You Need Downcasting

Build on your Creature hierarchy:

Creature[] party = {new Goblin(), new Dragon(), new Skeleton()};

for (Creature c : party) {
    // This works — attack() is in Creature
    System.out.println(c.attack());

    // This FAILS — breatheFire() is only in Dragon
    // c.breatheFire();  // COMPILE ERROR

    // Fix with downcasting:
    if (c instanceof Dragon) {
        Dragon d = (Dragon) c;
        System.out.println(d.breatheFire());
    }
}
ClassCastException: (Dragon)c compiles even if c isn't a Dragon — it crashes at RUNTIME. ALWAYS guard casts with instanceof. The AP exam will ask you to identify code that compiles but throws ClassCastException. Read the Downcasting docs Bug Hunt for the classic traps.

Build: Mixed Party Actions

Create a party of 5 Creature references with mixed subtypes. Write:

  • dragonCount() — uses instanceof to count Dragons
  • dragonAttack() — calls breatheFire() only on Dragons, regular attack() on others
  • dragonBattle() — finds all Dragons and has them fight each other (if 2+ exist)

Socratic Checkpoint

  1. Why does (Dragon)c compile even when c holds a Goblin? What does the compiler check?
  2. What's the difference between a compile error and a runtime error in the context of casting?
  3. Read the docs section on upcasting. Why is upcasting always safe but downcasting requires instanceof?

Checkpoint — before moving on, you must be able to:

  • Cast a parent reference to a child type with (ChildType)
  • Guard downcasts with instanceof
  • Explain why ClassCastException is a runtime error
