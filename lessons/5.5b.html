5.5b — Monster.java + Rewire DungeonCrawl

RPG Build ~30 min

Part 1: Monster.java — Same Pattern, New Class

Player.java taught you encapsulation. Now apply the same pattern to monsters. Open the Monster Builder (Student RPG Guide). Follow the specification.

Create Monster.java:

  • Private fields: name, hp, maxHp, ac, attackBonus, damageDie
  • Constructor takes all six. Clamp hp between 0 and maxHp.
  • Getters for all fields, isAlive(), takeDamage(), heal()
  • attack() — returns Dice.roll(damageDie)
  • toString() — "Goblin [HP: 10/15 AC:12 ATK:+2 d6]"

Socratic Checkpoint

  1. Player and Monster are nearly identical. What fields do they share? What fields are unique to each? If you were designing a game engine, what would you do to avoid duplicating code?
  2. Monster.attack() uses Dice.roll(). Why is this better than putting the dice formula directly in the attack method?

Part 2: Rewire DungeonCrawl

Open your DungeonCrawl.java. Replace every scattered player variable with a Player object and every scattered monster variable with a Monster object. 

// BEFORE
int playerHP = 25;
int playerAC = 14;
String playerName = "Hero";

// AFTER
Player player = new Player("Hero", 25, 25, 14, 12, 16, 10);

// Replace: playerHP -= damage     → player.takeDamage(damage)
// Replace: if (playerHP > 0)      → if (player.isAlive())
// Replace: playerHP += healAmt    → player.heal(healAmt)

Do the same for monsters:

// BEFORE
int monsterHP = 10;
int monsterAC = 12;
String monsterName = "Goblin";

// AFTER
Monster monster = new Monster("Goblin", 10, 10, 12, 2, 6);

// Replace: monsterHP -= dmg   → monster.takeDamage(dmg)
// Replace: if (monsterHP > 0) → if (monster.isAlive())
Watch Out: runCombat() currently takes individual int/String parameters. You'll need to update its signature to accept Player and Monster objects — or extract the values with getters. Which approach is cleaner? (The Monster Guide has the recommended approach.)

Run your dungeon crawler. Does everything still work? Fix any compile errors. The game logic should be identical — only the data storage changed.

Socratic Checkpoint

  1. Your old code had if (playerHP > 0) scattered everywhere. Now it's player.isAlive(). Why is the second version better even though it does the same thing?
  2. If you decided a player is dead at HP ≤ 0, where would you change that? Your old code vs your new code — how many file changes in each?
  3. Count the parameters to runCombat() before and after. How many lines of code disappeared from DungeonCrawl?

Checkpoint — before moving on, you must be able to:

  • Design a class spec and implement it from scratch
  • Refactor scattered primitives into encapsulated objects
  • Update method signatures to accept objects instead of primitives
  • Verify game logic is unchanged after refactoring
