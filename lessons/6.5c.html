6.5c — Wire Inventory Into Your RPG

RPG Build ~25 min

The Goal

You've built Item.java and FixedInventory.java in isolation. Now you integrate them into DungeonCrawl so loot from combat actually goes somewhere the player can use. This is the payoff: two files you built from scratch now power a real game feature.

Add Inventory to DungeonCrawl

Add a FixedInventory field to DungeonCrawl:

FixedInventory inventory = new FixedInventory(20);

After each combat victory, loot the monster:

inventory.add(new Item("Goblin Sword", 8, 25, "Common"));
inventory.add(new Item("Gold Coins", 1, 10, "Common"));

Add a "show inventory" command to your game loop. When the player types "inv", call inventory.display(). Show total gold value too.

Use Items in Combat

Before combat, let the player type "use Health Potion" to consume an item:

  • Check inventory.contains("Health Potion")
  • If found, call inventory.remove("Health Potion")
  • Call player.heal(15)

Add a "Strength Potion" that gives +3 attack for one combat. Where do you store "temporary buffs" — in the item, the player, or somewhere else?

Socratic Checkpoint

  1. FixedInventory has a capacity of 20. What happens when the 21st item drops? Is there a better way to handle this than silently losing items?
  2. You built Item as immutable. When a player "uses" a Health Potion, you don't change the potion — you remove it and call player.heal(). Why is this design cleaner than making Item mutable?
  3. Look at your DungeonCrawl main loop. How many features does it handle now: movement, combat, inventory, item usage. Is this getting unwieldy? What would you do to organize this in a bigger game?

Checkpoint — before moving on, you must be able to:

  • Wire a custom data structure into an existing game loop
  • Connect inventory operations (add/remove/contains) to game events
  • Explain the trade-off between mutable and immutable item design
