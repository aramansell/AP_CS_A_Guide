10.4a — Recursive Dungeon: Flood Fill & Pathfinding

RPG Build ~60 min

Your DungeonMap from Unit 8 is a grid, but you've been moving the player one step at a time. What if the player finds a magic item that reveals all connected floor tiles? Or a spell that finds the shortest path to the exit? These are naturally recursive problems — explore a tile, then explore all its neighbors, then their neighbors...

Open the Recursive Dungeon (Student RPG Guide). Follow the specification.

Part 1: Flood Fill

Add to DungeonMap:

// Starting from (row, col), mark all connected floor tiles as revealed
public void reveal(int row, int col, boolean[][] visited) {
    // Base case: off-map, wall, or already visited — stop
    if (row < 0 || row >= tiles.length || col < 0 || col >= tiles[0].length) return;
    if (tiles[row][col] == '#' || visited[row][col]) return;

    visited[row][col] = true;  // mark current

    // Recurse in all 4 directions
    reveal(row - 1, col, visited);  // up
    reveal(row + 1, col, visited);  // down
    reveal(row, col - 1, visited);  // left
    reveal(row, col + 1, visited);  // right
}

Call reveal(playerRow, playerCol, new boolean[numRows][numCols]) to find all connected floor space. Display visited tiles as '.' and unvisited as '?'.

Forgetting the Visited Check = Infinite Recursion: Without the visited array, reveal(0,0) calls reveal(1,0) which calls reveal(0,0) which calls reveal(1,0)... forever. The visited array breaks cycles. Also: check boundaries BEFORE indexing tiles[row][col]. Read the Recursive Dungeon docs for the complete flood-fill algorithm.

Part 2: Shortest Path

Add to DungeonMap:

// Return the fewest steps from (r1,c1) to (r2,c2), or -1 if unreachable
public int shortestPath(int r1, int c1, int r2, int c2, boolean[][] visited, int steps) {
    // Base cases
    if (out of bounds || wall || visited) return -1;
    if (r1 == r2 && c1 == c2) return steps;  // reached target!

    visited[r1][c1] = true;

    // Try all 4 directions, take the shortest valid path
    int up = shortestPath(r1-1, c1, r2, c2, visited, steps + 1);
    int down = shortestPath(r1+1, c1, r2, c2, visited, steps + 1);
    int left = shortestPath(r1, c1-1, r2, c2, visited, steps + 1);
    int right = shortestPath(r1, c1+1, r2, c2, visited, steps + 1);

    visited[r1][c1] = false;  // BACKTRACK so other paths can use this tile

    // Return the shortest non-negative result
    int min = Math.min(Math.min(up, down), Math.min(left, right));
    if (min == -1) {
        // All blocked? Check if any were valid (>= 0)
        if (up >= 0) min = up;
        else if (down >= 0) min = down;
        else if (left >= 0) min = left;
        else if (right >= 0) min = right;
    }
    return min;
}

Warning: this is the brute-force version. It tries EVERY path. For large maps, it's slow — but it's correct and it demonstrates backtracking.

Integrate Into Your RPG

Add two items to your inventory:

  • "Scroll of Revelation" — when used, calls reveal() from the player's position
  • "Compass" — when used, calls shortestPath() from player to exit marker 'E'

Add these as one-time-use items. Player can purchase them at the start of the dungeon.

Socratic Checkpoint

  1. Why does shortestPath need to set visited[r][c]=FALSE after exploring? What happens if you don't?
  2. Flood fill visits tiles in a depth-first pattern (explores a full path before trying the next direction). What would breadth-first look like? Which is better for shortest-path?
  3. Read the docs section on why we return -1. How do you distinguish "tried all paths, none work" from "haven't tried yet"?

Checkpoint — before moving on, you must be able to:

  • Implement recursive flood fill with visited tracking
  • Implement recursive shortest-path with backtracking
  • Explain why backtracking requires resetting visited flags
  • Integrate recursive algorithms into the RPG game loop
