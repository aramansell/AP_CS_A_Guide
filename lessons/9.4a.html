9.4a — Interfaces: Contracts, Not Parents

Discovery Lab ~25 min

Inheritance means IS-A. A Goblin IS-A Creature. But what about IS-ABLE-TO? A Goblin CAN-FIGHT. A Dragon CAN-FIGHT. A Player CAN-FIGHT. None of them share a parent that makes sense for "fighting" — but they all need attack() and takeDamage(). An interface is a contract: "any class that implements me MUST provide these methods." No fields, no constructors, no inheritance — just method signatures.

Experiment: Your First Interface

Create Combatant.java (an interface, NOT a class):

public interface Combatant {
    int attack();
    void takeDamage(int damage);
    boolean isAlive();
    String getName();
}

Now make Creature implement it:

public class Creature implements Combatant {
    // Creature already HAS these methods — no changes needed!
    // But now you can also make Player implement Combatant
}

Make Player implement Combatant too. Now write polymorphic code:

Combatant[] fighters = new Combatant[3];
fighters[0] = new Goblin();
fighters[1] = new Dragon();
fighters[2] = new Player("Hero", 100, 15);

for (Combatant c : fighters) {
    System.out.println(c.getName() + " attacks: " + c.attack());
}

Player and Creature share NO parent class (well, Object). But they both implement Combatant, so they're both valid here.

Interface ≠ Parent Class: An interface CANNOT have instance variables, constructors, or method bodies (before Java 8). It's pure method signatures. A class can implement MULTIPLE interfaces. Read the Comparison Interfaces docs for the full interface rules and the Comparable pattern.

Build: Comparable

Make Item implement Comparable<Item>:

public class Item implements Comparable<Item> {
    // ... existing fields and methods ...

    @Override
    public int compareTo(Item other) {
        return this.value - other.value;  // sort by value ascending
    }
}

Now Collections.sort(inventory) works. Try it with your DynamicInventory. Then change compareTo to sort by name alphabetically instead.

Socratic Checkpoint

  1. Could you have used an abstract class instead of Combatant? What's the difference?
  2. Why can you implement multiple interfaces but only extend one class?
  3. Read the docs section on Comparator vs Comparable. When would you use one over the other?

Checkpoint — before moving on, you must be able to:

  • Define an interface with method signatures
  • Implement an interface on multiple unrelated classes
  • Write polymorphic code using interface references
  • Implement Comparable for natural sort order
