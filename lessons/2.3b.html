2.3b — Scanner: Character Creation Terminal

Basic Application ~25 min

You know how to read one value. But RPG character creation needs many: name, class, stats, weapon. Today you build a terminal-based character creator that asks the player a series of questions and assembles a character sheet.

Buffer trap refresher: Every time you mix nextInt() or nextDouble() with nextLine(), the leftover newline will eat your next String read. You MUST add an extra nextLine() to consume it. If you're not sure why, re-read the Scanner docs — buffer trap section.

Build: CharacterCreator.java

Create a program that asks the player these questions in order:

  1. "Enter your character name:" — read with nextLine()
  2. "Choose a class (warrior/mage/rogue):" — read with nextLine()
  3. "Enter strength (1-20):" — read with nextInt()
  4. "Enter dexterity (1-20):" — read with nextInt()
  5. "Enter intelligence (1-20):" — read with nextInt()
  6. "Enter starting gold:" — read with nextDouble()

Then print a formatted character sheet:

═══════════════════════════════
  CHARACTER SHEET
═══════════════════════════════
  Name: Thorn
  Class: rogue
  STR: 14  DEX: 18  INT: 10
  Gold: 25.5
═══════════════════════════════

Bug Hunt: Find the Buffer Traps

There are at least THREE buffer traps in this sequence. Find them before you run the program. Write down which lines will break and why. Then run it — were you right?

Fix every trap. Your program must read all six values correctly with no skips or crashes.

Socratic Checkpoint

  1. You read strength, dexterity, and intelligence one after another. Did any of those cause buffer traps? Why or why not?
  2. After nextDouble() for gold, you ask for the next input. What trap appears? Why is it the same pattern as nextInt()?
  3. Why does nextLine() after nextDouble() consume the newline, but nextInt() after nextInt() works fine?

Checkpoint — before moving on, you must be able to:

  • Chain multiple Scanner reads without buffer traps
  • Identify every location where nextInt/nextDouble is followed by nextLine
  • Format multi-line output with borders and aligned columns
  • Explain why consecutive nextInt() calls DON'T need buffer fixes
