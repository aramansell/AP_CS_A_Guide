10.4b — Shortest Path & Wire It All In

RPG Build ~30 min

Part 1: Recursive Shortest Path

Flood fill explores all connected tiles. Now find the shortest route from the player to the exit. This requires backtracking — try a path, mark tiles as you go, then unmark them when you backtrack so other paths can use them. This is the trickiest recursive pattern in the course.

Open the Recursive Dungeon (Student RPG Guide). Follow Part 2: Shortest Path.

Add to DungeonMap:

// Return fewest steps from (r1,c1) to (r2,c2), or -1 if unreachable
public int shortestPath(int r1, int c1, int r2, int c2,
                        boolean[][] visited, int steps) {
    // Base case: out of bounds or wall or visited
    if (r1 < 0 || r1 >= tiles.length || c1 < 0 || c1 >= tiles[0].length)
        return -1;
    if (tiles[r1][c1] == '#' || visited[r1][c1])
        return -1;
    // Base case: reached target!
    if (r1 == r2 && c1 == c2)
        return steps;

    visited[r1][c1] = true;

    // Try all 4 directions
    int up    = shortestPath(r1-1, c1, r2, c2, visited, steps + 1);
    int down  = shortestPath(r1+1, c1, r2, c2, visited, steps + 1);
    int left  = shortestPath(r1, c1-1, r2, c2, visited, steps + 1);
    int right = shortestPath(r1, c1+1, r2, c2, visited, steps + 1);

    visited[r1][c1] = false;  // BACKTRACK — release this tile

    // Find the shortest valid path
    int min = -1;
    if (up    >= 0 && (min == -1 || up    < min)) min = up;
    if (down  >= 0 && (min == -1 || down  < min)) min = down;
    if (left  >= 0 && (min == -1 || left  < min)) min = left;
    if (right >= 0 && (min == -1 || right < min)) min = right;

    return min;
}
Backtracking is Critical: After exploring all 4 directions, you MUST set visited[r1][c1] = false. Without this, the first path found blocks all other paths from using that tile. The shortest path might route through a tile that a longer path visited first. If you forget the backtracking line, you'll get a valid path but not necessarily the shortest one.

Socratic Checkpoint

  1. Why does shortestPath need visited[r][c] = false at the end but flood fill doesn't? What's different about their goals?
  2. Read the docs section on why we return -1. How do you distinguish "tried all paths, none work" from "haven't tried yet"?
  3. This algorithm tries EVERY path. For a 15×15 map, that could mean thousands of recursive calls. It's correct but slow. What could you do to make it faster while still being correct?

Part 2: Wire Into Your RPG

Add two items to your inventory system:

  • "Scroll of Revelation" — when used, calls reveal() from the player's position and displays the fog-of-war map. One-time use.
  • "Compass" — when used, calls shortestPath() from player to the exit marker 'E' and prints: "The exit is 12 steps away." One-time use.

Add a "use compass" command to your game loop:

if (command.equals("compass") && inventory.contains("Compass")) {
    int steps = map.shortestPath(playerRow, playerCol,
                    exitRow, exitCol,
                    new boolean[rows][cols], 0);
    if (steps >= 0)
        System.out.println("The compass needle spins. Exit: " + steps + " steps away.");
    else
        System.out.println("The compass spins wildly — no path found!");
    inventory.remove("Compass");
}

Tinker: Dijkstra's Distance Map

ShortestPath is brute-force and slow. For extra challenge: implement a distanceMap(int startRow, int startCol) method that returns an int[][] where each cell contains the distance from the start. Use a queue (not recursion) for breadth-first search. This is Dijkstra's algorithm for unweighted graphs — and it's MUCH faster. Read the recursive traversal docs for the BFS pattern.
