3.4a — Comparing Objects in Conditions

Discovery Lab ~20 min

You already know the golden rule from Unit 2: primitives use ==, objects use .equals(). Now you need to use that rule inside conditions.

Experiment: The == Trap

Create CompareTest.java. Run this code:

int a = 100;
int b = 100;
System.out.println(a == b);     // what do you expect?

String s1 = "hello";
String s2 = "hello";
System.out.println(s1 == s2);   // what do you expect?
System.out.println(s1.equals(s2)); // what about this?

String s3 = new String("hello");
System.out.println(s1 == s3);   // NOW what happens?

Write down your predictions for each one BEFORE running. Then run it. Which results surprised you?

Socratic Checkpoint

  1. Why did s1 == s2 print true, but s1 == s3 print false — even though they both contain "hello"?
  2. Open the Comparing Objects docs. Find the section on null-safe comparisons. What's the difference between name.equals("Frodo") and "Frodo".equals(name)? Why does one crash when name is null and the other doesn't?
  3. How does short-circuit evaluation (from 3.2a) help when writing safe null checks before .equals()?

Build: Combat Command Parser

Write a program that uses Scanner to read a command from the player (attack, defend, potion, flee). Use if/else if chains with .equals() to print the appropriate action.

Test it. Then try typing "Attack" (with a capital A). What happens? Read the docs to learn how to make your parser case-insensitive using string methods you learned in Unit 2.

Explore: compareTo

The Comparing Objects docs have a section on compareTo. Read it. Then write a program that compares two strings alphabetically and prints which one comes first. What value does compareTo return? What do negative, zero, and positive mean?

Explore: switch Statement

The docs also show a switch statement — an alternative to long if/else chains for comparing one value against many options. Read that section. Then find the Bug Hunt about the fall-through trap. Rewrite your Combat Command Parser using switch instead of if/else if. What does the break keyword do? What happens if you forget it?

Checkpoint — before moving on, you must be able to:

  • Use .equals() for String comparisons in conditions
  • Explain why == is unreliable for objects
  • Write null-safe object comparisons
  • Use compareTo for ordering
  • Implement a switch statement with proper break usage
