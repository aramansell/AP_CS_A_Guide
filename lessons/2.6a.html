2.6a — Wrapper Classes: Primitives in Disguise

Discovery Lab ~30 min

You've used int, double, and boolean since day one. But sometimes Java needs them to act like objects. Enter wrapper classes: Integer, Double, Boolean, Character.

Why do wrappers exist? Later this year, you'll use data structures like ArrayList that can only hold objects — not primitives. You can't say ArrayList<int>. You say ArrayList<Integer>. Today you learn what wrappers are and the traps that come with them.

Launch — What Is This?

Create WrapperLab.java and run this:

System.out.println("Biggest int: " + Integer.MAX_VALUE);
System.out.println("Smallest int: " + Integer.MIN_VALUE);
System.out.println("Biggest double: " + Double.MAX_VALUE);

Socratic Checkpoint

  1. Integer.MAX_VALUE — is this a method call or a constant?
  2. Why is it on Integer (capital I) rather than int (lowercase)?
  3. What happens if you add 1 to Integer.MAX_VALUE? Try it. This is called integer overflow.

Section 1: parseInt — String to Number

The most useful wrapper trick: turning a String into a number. 

String numText = "42";
int actualNumber = Integer.parseInt(numText);
System.out.println(actualNumber + 8);  // 50 — math works!

String decimalText = "3.14";
double pi = Double.parseDouble(decimalText);
System.out.println(pi * 2);  // 6.28

But beware — garbage in, crash out:

String bad = "hello";
int n = Integer.parseInt(bad);  // NumberFormatException — CRASH!

Tinker Challenge: Parse and Sum

Scanner's nextInt() reads numbers directly. So why would you ever need Integer.parseInt()? Here's a situation: what if you already have a String full of numbers and need to do math with them?

Write a program that reads a line of text, splits it on commas, and converts each piece to an integer. Then sum them. You'll need to figure out how to split a String into pieces — open the String docs and find a method that does that. 

Enter numbers separated by commas: 10,25,13,8
Sum: 56

Section 2: Autoboxing — Java Does It For You

Most of the time, Java converts automatically between primitives and wrappers:

Integer wrapped = 42;    // autoboxing: int → Integer
int unwrapped = wrapped; // unboxing: Integer → int

// These are DIFFERENT types, but Java handles the conversion.
// You usually don't have to think about it.

THE CACHING TRAP — This One Is Famous

Create a file called CacheTrap.java and run this:

Integer a = 100;
Integer b = 100;
Integer c = 200;
Integer d = 200;

System.out.println(a == b);  // predict: ____
System.out.println(c == d);  // predict: ____
System.out.println(a.equals(b)); // predict: ____
System.out.println(c.equals(d)); // predict: ____

Were the results surprising? If two of the lines didn't do what you expected, you've just discovered something important. Your job: figure out why 100 works one way but 200 works differently. Open the Wrapper Classes docs and find the section on Integer comparison. Then answer: what method should you ALWAYS use to compare wrapper objects, and why?

Socratic Checkpoint

  1. What method should you ALWAYS use to compare Integer objects?
  2. Is this caching trap the same as the == vs equals() problem with Strings? Why or why not?
  3. If you wrote int a = 200; int b = 200; (primitives, not wrappers), would a == b work correctly?

Section 3: Unboxing Dangers

Bug Hunt 1: The Null Trap

Integer score = null;
int finalScore = score;  // What happens?

Run this. What error do you get? Why can a wrapper be null but a primitive cannot? Check the Wrapper docs — look for the section that explains what happens when Java tries to convert null into a primitive. What's the fix?

Bug Hunt 2: Wrong Parser

String input = "10.5";
int value = Integer.parseInt(input);  // What happens?

"10.5" has a decimal point. Why does Integer.parseInt fail here? Which wrapper class's parser SHOULD you use for decimal numbers? Check the Wrapper docs and find the right one.
