3.6c — Combat Engine: Full Encounter

RPG Build ~30 min

You built the engine. Now use it. Create a CombatEncounter.java with a main method that stages a complete fight between your character and a monster — from first roll to final blow.

Requirements

  • Hard-code stats for a hero and a monster (attack bonuses, AC, damage dice, HP, names)
  • Call combatRound() repeatedly until either the player or the monster reaches 0 HP. (Yes, this means you'll need a loop — the Unit 4 docs have what you need. Just a basic while loop. This is your preview of what's coming.)
  • After each round, print who's winning based on remaining HP
  • When the fight ends, print a victory or defeat message
Loop Preview: You haven't learned loops formally yet. Here's enough to get you through this encounter: 
while (monsterHP > 0 && playerHP > 0) {
    // your combat round code here
}
This keeps running combat rounds until someone's HP drops to 0 or below.

Socratic Checkpoint — Unit 3 Reflection

  1. Trace through your full encounter. Where does the if/else chain decide hit vs miss? Where does boolean logic combine conditions? Where does De Morgan's law make a condition cleaner?
  2. Your Combat.java methods take many parameters. Why pass them in rather than storing them as instance variables? (Hint: compare to how Character.java stores its stats.)
  3. What makes a good combat encounter? Is it more fun when the player is slightly stronger than the monster, or slightly weaker? Tweak your stats and find the sweet spot.
  4. In Unit 5, you'll move these methods into Player and Monster classes. How would that change the architecture?

Tinker: Commander Mode

Before the fight starts, use Scanner to let the player pick their action. Offer three choices using .equals():

  • "attack" — fight normally
  • "defend" — reduce incoming damage by half (you'll need a new variable to track this)
  • "flee" — attempt to escape (50% chance using a d20 roll)

This combines EVERYTHING from Unit 3: if/else chains, boolean logic, comparing objects with .equals(), Scanner input, and your Dice class.

Checkpoint — before moving on, you must be able to:

  • Assemble multiple methods into a complete interactive experience
  • Track changing state (HP, status) across multiple method calls
  • Use a basic while loop to repeat combat rounds
  • Combine all Unit 3 concepts (if/else, boolean logic, comparisons) in one program
  • Explain the architecture: Dice → Combat → CombatEncounter
