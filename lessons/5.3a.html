5.3a — Accessors & Mutators: Controlled Access

Discovery Lab ~25 min

Your Weapon class has private String name. The private keyword means only code inside Weapon.java can touch it. But what if GameLoop needs to read the weapon's damage? It can't reach sword.damage directly. You need accessor methods (getters) to expose data safely, and mutator methods (setters) to change it with validation.

Experiment: Public vs Private

Take your Monster.java from 5.2a. Change private int hp to public int hp. Now in your test class, write:

Monster goblin = new Monster("Goblin", 10, 12, 3);
goblin.hp = -999;  // HAHAHA IMMORTAL GOBLIN
System.out.println(goblin);

This compiles. This is a disaster. Change it back to private. Now add:

public int getHP() { return hp; }
public void setHP(int newHP) {
    if (newHP < 0) newHP = 0;
    hp = newHP;
}

Try goblin.hp = -999 again — it won't compile. Try goblin.setHP(-999) — the setter clamps it to 0.

Boolean Naming Trap: Read about proper getter naming in the accessors & mutators docs. If you have private boolean alive, the getter should be isAlive(), not getAlive(). This is a Java convention that tools and frameworks depend on.

Build: Player HP System

Create Player.java. It needs:

  • private instance variables: name, hp, maxHp, ac, strength
  • Constructor that sets all values. Clamp hp to maxHp.
  • getHP(), getMaxHP(), getName(), getAC() — accessors
  • takeDamage(int amount) — subtracts from hp, clamps to 0
  • heal(int amount) — adds to hp, clamps to maxHp
  • isAlive() — returns true if hp > 0
  • toString() — shows name, HP/maxHP, AC

Test: create a player, damage them, heal them, print after each step.

Socratic Checkpoint

  1. Why use takeDamage() instead of setHP(getHP() - damage)? Which one is safer? Which one reads better?
  2. If hp is private and you have getHP(), why can't someone write goblin.getHP() = 10? What's the difference between reading a value and assigning to a method call?
  3. Look at the docs section on immutability. What does it mean for a class to be immutable? Is your Player immutable? Should it be?

Checkpoint — before moving on, you must be able to:

  • Write getters (accessors) and setters (mutators) with proper naming
  • Explain why instance variables should be private
  • Add validation logic inside mutators (clamping, bounds checking)
  • Use is prefix for boolean getters
