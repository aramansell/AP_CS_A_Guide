10.3b — Recursive Fibonacci (Build Together)

Type: Basic Application. The teacher builds, you follow along. The classic recursive function — elegant, simple, and deceptively slow. You'll watch it blow up and learn why recursion isn't always the answer.

Basic Application ~30 min

The Goal

Write a recursive Fibonacci function. Watch it work beautifully for small numbers. Then push it to n=50 and witness why exponential recursion is a trap.

fib(0) = 0
fib(1) = 1
fib(2) = 1
fib(3) = 2
fib(4) = 3
fib(5) = 5
fib(10) = 55
fib(20) = 6765
fib(40) = 102334155
fib(50) = ... waiting ... waiting ...

Reference: Fibonacci & Efficiency — full reference.

Iteration 1: The Recursive Formula

Create Fibonacci.java. The mathematical definition:

  • fib(0) = 0
  • fib(1) = 1
  • fib(n) = fib(n-1) + fib(n-2)
public class Fibonacci {
    public static long fib(int n) {
        if (n == 0) return 0;      // base case 1
        if (n == 1) return 1;      // base case 2
        return fib(n - 1) + fib(n - 2);  // recursive case
    }

    public static void main(String[] args) {
        for (int i = 0; i <= 10; i++) {
            System.out.println("fib(" + i + ") = " + fib(i));
        }
    }
}

Socratic Checkpoint

  1. Why are there TWO base cases? What would happen with only one (n == 0)?
  2. Trace fib(4) by hand. Draw the call tree. How many times does fib(2) get called?
  3. What's different about long vs int here? Why did we use long?

Iteration 2: The Exponential Explosion

Try fib(40). It works but feels slow. Try fib(45). You'll wait. Try fib(50). You'll wait a very long time. Why?

Bug Hunt — Not a Bug, a Design Flaw

The algorithm is correct but does massive redundant work. Count how many times fib(2) is called during fib(10). Then count for fib(20). The pattern is exponential: O(2ⁿ).

Add a counter to your class:

static int calls = 0;
// inside fib(): calls++;

Run fib(20) and print calls. How many calls did it take to compute a single number?

Iteration 3: The Fix (Memoization Preview)

The fix is to remember results you've already computed. This is called memoization. We'll build it properly in the Expand application, but for now, see if you can figure out the pattern:

static long[] memo = new long[100];  // stores computed fib(n)

public static long fibMemo(int n) {
    if (n == 0) return 0;
    if (n == 1) return 1;
    if (memo[n] != 0) {      // already computed?
        return memo[n];
    }
    memo[n] = fibMemo(n - 1) + fibMemo(n - 2);
    return memo[n];
}

Run fibMemo(50). Instant. Why? How many recursive calls does it make now vs. the original?

Socratic Checkpoint

  1. The memoized version calls each n EXACTLY ONCE. Why does if (memo[n] != 0) guarantee that?
  2. Where else in programming do we trade memory for speed? Is this always the right trade-off?
