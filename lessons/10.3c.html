10.3c — Recursive Palindrome Checker

Type: Expand Application. Fibonacci had a formula you could just translate. Now design recursion from scratch — what's the base case? What's the recursive step? The palindrome problem forces you to think like a recursive designer.

Expand Application ~25 min

The Goal

Write a recursive method that checks if a String is a palindrome (reads the same forward and backward). Then compare it to the iterative version from Unit 4. Which is clearer?

"racecar"     => true
"hello"       => false
"abba"        => true
"a"           => true (base case!)
""            => true (base case!)
"A man a plan a canal Panama"  => true (ignoring spaces/case)

Build It: isPalindrome()

Create PalindromeChecker.java. Think about the recursive definition of a palindrome:

  • An empty string is a palindrome (base case)
  • A single character is a palindrome (base case)
  • A string is a palindrome if first == last AND the middle is a palindrome
public class PalindromeChecker {
    public static boolean isPalindrome(String s) {
        // Base cases
        if (s.length() <= 1) {
            return true;
        }

        // Compare first and last characters
        char first = s.charAt(0);
        char last = s.charAt(s.length() - 1);

        if (first != last) {
            return false;
        }

        // Recursive step: check the middle
        String middle = s.substring(1, s.length() - 1);
        return isPalindrome(middle);
    }

    public static void main(String[] args) {
        System.out.println(isPalindrome("racecar")); // true
        System.out.println(isPalindrome("hello"));   // false
        System.out.println(isPalindrome("abba"));    // true
    }
}

Socratic Checkpoint

  1. Trace isPalindrome("abba"). Draw the call stack. What's the sequence of recursive calls?
  2. Why does s.length() <= 1 cover both the empty string AND single-character cases? How does an even-length palindrome like "abba" eventually reach this base case?
  3. What does substring(1, s.length()-1) return for "abba"? For "racecar"? Trace it.

Bug Hunt

The code above fails for "Racecar" (capital R). Fix it so case doesn't matter. Then fix it so spaces don't matter (for multi-word palindromes). Which approach is simpler: cleaning the string BEFORE recursion, or handling it INSIDE each recursive call?

Recursive vs. Iterative

In 4.5c you built an iterative palindrome checker (loop from both ends). Now you have both. Compare them:

// Iterative (Unit 4)
for (int i = 0; i < s.length() / 2; i++) {
    if (s.charAt(i) != s.charAt(s.length() - 1 - i)) {
        return false;
    }
}
return true;

// Recursive (this unit)
if (s.length() <= 1) return true;
if (s.charAt(0) != s.charAt(s.length() - 1)) return false;
return isPalindrome(s.substring(1, s.length() - 1));

Socratic Checkpoint

  1. Which version is more efficient? Why? (Think about String creation: substring() creates a new String object every recursive call.)
  2. Which version is easier to understand? Why?
  3. When would you choose recursion over iteration, even if it's less efficient?

Extend It (pick one)

  • Write a recursive method that reverses a string (same pattern: first char goes to the end).
  • Write a recursive method that counts vowels in a string.
  • Write a recursive method that finds the longest palindromic substring within a longer string.
