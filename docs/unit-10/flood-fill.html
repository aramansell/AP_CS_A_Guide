Student RPG Guide: Area-of-Effect Spells (Flood Fill)

Fireball, poison gas, holy light. Any spell that hits ALL enemies in a connected region uses flood fill. Unlike chain lightning (which has hop limits), flood fill explores everything reachable without crossing walls. Two files: AoESpell.java and AoESpellRunner.java.

Flood Fill vs Chain Lightning

FeatureChain LightningFlood Fill / Fireball
Hop limitYes — maxChainsNo — fills everything connected
Damage per targetHalves each hopSame damage to all enemies
Empty cellsSkipped (only hits characters)Traversed through (spreads fireball through corridors)
Stopping conditionsBounds, null/dead, visited, max chainsBounds, walls, visited only
Visited cellsNever un-visitedNever un-visited (same)
Traversal styleBFS-like (hit current, recurse outward)DFS (recurse into neighbors before processing siblings)

The critical difference: flood fill recurses even into EMPTY cells (grid[row][col] == null). Empty cells get marked visited, then their neighbors are checked. This is how a fireball spreads down a corridor to hit enemies in the next room.

ParameterTypePurpose
gridGameCharacter[][]Characters on the battle map (can be null)
terrainboolean[][]True = wall (impassable). Same dimensions as grid.
row, colintWhere the spell originates
damageintDamage dealt to EVERY enemy reached (no halving)
visitedboolean[][]Prevents revisiting. Same dimensions as grid.

Returns int — how many enemies were damaged.

Base Cases (Order Matters!)

        // (1) Out of bounds
        if (row < 0 || row >= grid.length
            || col < 0 || col >= grid[0].length)
            return 0;

        // (2) Wall — impassable
        if (terrain[row][col])
            return 0;

        // (3) Already visited
        if (visited[row][col])
            return 0;

Recursive Step

        // Mark visited (even if empty — the FIRE went through here)
        visited[row][col] = true;

        int count = 0;

        // If there's an alive enemy here, damage it
        if (grid[row][col] != null && grid[row][col].isAlive()) {
            grid[row][col].takeDamage(damage);
            System.out.println("Fireball hits "
                + grid[row][col].getName()
                + " for " + damage + "!");
            count = 1;
        }

        // Recursively flood fill ALL 4 neighbors
        // (even if this cell was empty — the spell keeps spreading)
        count += floodFill(grid, terrain, row - 1, col, damage, visited);
        count += floodFill(grid, terrain, row + 1, col, damage, visited);
        count += floodFill(grid, terrain, row, col - 1, damage, visited);
        count += floodFill(grid, terrain, row, col + 1, damage, visited);

        return count;
    }
}

Why recurse into empty cells? An empty cell is a corridor, an open room, a path. The fireball doesn't stop at empty space — it keeps going until it hits walls. If we only recursed on cells with enemies, fireball couldn't travel from one room to another.

FILE 2: AoESpellRunner.java

NEW FILE. Tests flood fill on a grid with walls separating two rooms.

public class AoESpellRunner {
    public static void main(String[] args) {

        // 6x6 grid
        GameCharacter[][] grid = new GameCharacter[6][6];

        // Enemies in the LEFT room
        grid[1][1] = new Monster("Goblin A", 12, 10, 6);
        grid[1][2] = new Monster("Goblin B", 12, 10, 6);
        grid[2][1] = new Monster("Orc", 20, 12, 8);

        // Enemy in the RIGHT room — separated by a wall!
        grid[4][3] = new Monster("Troll", 30, 14, 10);

        // Terrain: true = wall
        boolean[][] terrain = new boolean[6][6];

        // Outer walls (border)
        for (int i = 0; i < 6; i++) {
            terrain[0][i] = true;   // top
            terrain[5][i] = true;   // bottom
            terrain[i][0] = true;   // left
            terrain[i][5] = true;   // right
        }

        // Inner dividing wall at row 3 (separates left room from right room)
        terrain[3][0] = true;
        terrain[3][1] = true;
        terrain[3][2] = true;
        terrain[3][4] = true;
        terrain[3][5] = true;
        // Note: terrain[3][3] is FALSE — that's a gap/passage? No.
        // Actually, keep it a wall too for a clean separation:
        terrain[3][3] = true;

        // Display initial state
        System.out.println("=== INITIAL GRID ===");
        for (int r = 0; r < grid.length; r++) {
            for (int c = 0; c < grid[r].length; c++) {
                if (terrain[r][c]) {
                    System.out.print("###  ");
                } else if (grid[r][c] != null) {
                    System.out.print(grid[r][c].getName().substring(0,3) + "  ");
                } else {
                    System.out.print(".    ");
                }
            }
            System.out.println();
        }

        // Cast fireball at (1,1)
        System.out.println("\n=== CASTING FIREBALL ===");
        boolean[][] visited = new boolean[6][6];
        int hit = AoESpell.floodFill(grid, terrain, 1, 1, 10, visited);
        System.out.println("Enemies hit by fireball: " + hit);

        // Display final state
        System.out.println("\n=== FINAL GRID ===");
        for (int r = 0; r < grid.length; r++) {
            for (int c = 0; c < grid[r].length; c++) {
                if (terrain[r][c]) {
                    System.out.print("###  ");
                } else if (grid[r][c] != null) {
                    GameCharacter gc = grid[r][c];
                    String status = gc.isAlive() ? "alive" : "DEAD";
                    System.out.print(gc.getName().substring(0,3) + ":" + status + " ");
                } else {
                    System.out.print(".    ");
                }
            }
            System.out.println();
        }
    }
}

Expected: 3 enemies hit (Goblin A, Goblin B, Orc). Troll is UNHARMED — the wall at row 3 blocks flood fill from reaching row 4.

Integrate Into Your RPG

BeforeAfter
Chain lightning only (hop-limited)Fireball covers entire connected rooms
Spells ignore terrainWalls block spell spread
Single-target combatArea-of-effect spells across the dungeon grid

Flood fill also powers non-combat features: reachability checks ("can the player reach this treasure?"), room detection ("is the exit in a connected region?"), and map validation ("is this dungeon solvable?").
