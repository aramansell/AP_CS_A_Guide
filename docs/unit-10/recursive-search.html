Recursive Search

You've searched arrays with for loops since Unit 6. Now: the same problem, solved with recursion. Each call checks one element; if it's not the target, the call passes the rest of the array to itself.

Recursive Linear Search

// Public API — clean, no index parameter exposed
public static int search(int[] arr, int target) {
    return searchHelper(arr, target, 0);
}

// Private helper does the real work
private static int searchHelper(int[] arr, int target, int index) {
    if (index >= arr.length) return -1;     // BASE: fell off the end
    if (arr[index] == target) return index; // BASE: found it!
    return searchHelper(arr, target, index + 1); // RECURSIVE: check next
}

search([10, 20, 30, 40], target=30) → 2
search([10, 20, 30, 40], target=99) → -1

The Helper Pattern: The public method has a clean signature — just the array and target. The private helper adds an index parameter that tracks progress. This pattern is everywhere in recursive code.

Tracing a Search

Searching for 30 in [10, 20, 30, 40]:

searchHelper(arr, target=30, index=0)
  arr[0]=10 → not 30, call index=1
    searchHelper(arr, target=30, index=1)
      arr[1]=20 → not 30, call index=2
        searchHelper(arr, target=30, index=2)
          arr[2]=30 → FOUND! return 2
        → return 2
      → return 2
    → return 2

Once the answer is found deep in the recursion, it bubbles all the way back up unchanged. Each level just passes the return value along.

Searching ArrayList

Same pattern, different data structure:

public static int searchList(ArrayList<String> list, String target) {
    return searchListHelper(list, target, 0);
}

private static int searchListHelper(ArrayList<String> list,
                                     String target, int index) {
    if (index >= list.size()) return -1;
    if (list.get(index).equals(target)) return index;
    return searchListHelper(list, target, index + 1);
}

Key difference: .equals() instead of == for String comparison. Same logic otherwise.

Recursive vs Iterative Search

Iterative (loop)Recursive
Progress i++ each iteration index + 1 each call
Stop condition Loop condition: i < arr.length Base case: index >= arr.length
Found target return i; return index;
Memory Constant — no stack growth One stack frame per element (O(n))

For linear search, iteration is better — same logic, less memory. Recursion wins with problems that have natural tree structure, like binary search on a sorted array, or flood fill on a grid.

Recursive Binary Search (Challenge)

Binary search cuts the search space in half each step. This IS naturally recursive — each call searches a smaller subarray. 

public static int binarySearch(int[] arr, int target) {
    return binarySearchHelper(arr, target, 0, arr.length - 1);
}

private static int binarySearchHelper(int[] arr, int target,
                                       int left, int right) {
    if (left > right) return -1;           // BASE: empty range
    int mid = left + (right - left) / 2;
    if (arr[mid] == target) return mid;    // BASE: found it
    if (arr[mid] > target)
        return binarySearchHelper(arr, target, left, mid - 1); // go left
    else
        return binarySearchHelper(arr, target, mid + 1, right); // go right
}

Binary search is O(log n) — cuts the search space in half each call. Even on a million-element array, it takes about 20 recursive calls. Compare to linear search's million iterations.

Socratic Checkpoint

1. The Wrapper Pattern API

We notice that both recursive linear search and recursive binary search use a public method (e.g., binarySearch(int[] arr, int target)) that delegates to a private helper method (e.g., binarySearchHelper(...)). Why do we hide the index parameters from the outside world? Why is forcing a caller to type search(myArray, target, 0) instead of just search(myArray, target) considered a poor API design practice?

2. Binary Search Stack Execution

Suppose you call binarySearch for the target value 7 on the sorted array arr = {1, 3, 5, 7, 9, 11}. Trace the values of left, right, and mid for each step of the recursive helper. How many total recursive stack frames are pushed onto the call stack before returning the index?

Bug Hunt: The Infinite Subarray Trap

A developer writes a custom recursive binary search helper. However, when they test it with a target that doesn't exist in the array (e.g., searching for 2 in {1, 3, 5}), the program hangs and crashes with a StackOverflowError.

private static int badBinarySearchHelper(int[] arr, int target, int left, int right) {
    if (left > right) return -1;
    
    int mid = left + (right - left) / 2;
    if (arr[mid] == target) return mid;
    
    if (arr[mid] > target) {
        return badBinarySearchHelper(arr, target, left, mid); // Potential issue
    } else {
        return badBinarySearchHelper(arr, target, mid, right); // Potential issue
    }
}

Question: Spot the error! Why does passing mid instead of mid - 1 or mid + 1 lead to an infinite recursive loop? Show the exact boundary conditions under which left and right will freeze and refuse to move toward the base case.
