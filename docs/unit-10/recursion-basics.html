Recursion Basics

Recursion is when a method calls itself to solve a smaller version of the same problem. It's like a Russian nesting doll — each doll contains a smaller doll, until you reach the tiniest solid one at the center.

The Two Required Parts

Every recursive method MUST have exactly two things:

  1. Base case — when the problem is trivially small, stop. This prevents infinite recursion and StackOverflowError.
  2. Recursive call — call the same method on a smaller/simpler problem that moves toward the base case. 
public static void countdown(int n) {
    if (n <= 0) {                    // BASE CASE: stop here
        System.out.println("Go!");
        return;
    }
    System.out.print(n + "... ");    // do one step
    countdown(n - 1);               // RECURSIVE: smaller number
}

countdown(3) prints: 3... 2... 1... Go!

The Call Stack

Every time a method is called, Java pushes a stack frame — a block of memory holding that call's parameters and local variables. When the method returns, its frame is popped off.

During recursion, the stack grows: 

countdown(3)
  → calls countdown(2)
      → calls countdown(1)
          → calls countdown(0)
              → base case! returns
          → returns
      → returns
  → returns

Each call has its own copy of n. They don't interfere with each other. When a call returns, its n is gone.

Why "StackOverflowError"? The call stack has a maximum size. If recursion never hits a base case, it keeps pushing frames until the stack overflows — Java throws StackOverflowError and your program crashes.

Factorial — The Classic Example

The factorial of n, written n!, is the product of all integers 1 through n: 

5! = 5 × 4 × 3 × 2 × 1 = 120

Recursive definition: n! = n × (n-1)!, with 0! = 1 as the base case.

public static int factorial(int n) {
    if (n <= 1) return 1;           // BASE: 0! = 1, 1! = 1
    return n * factorial(n - 1);    // n! = n × (n-1)!
}

Tracing factorial(4):

factorial(4)
  → 4 × factorial(3)
          → 3 × factorial(2)
                  → 2 × factorial(1)
                          → 1          (base case!)
                  → 2 × 1 = 2
          → 3 × 2 = 6
  → 4 × 6 = 24

Notice: the multiplication waits for the recursive call to return. The 4 × sits on the stack until factorial(3) finishes.

The Trust Fall

The hardest mental shift with recursion is trusting the recursive call. When you write return n * factorial(n - 1), you must believe that factorial(n - 1) correctly returns (n-1)!. You don't trace into it — you trust it.

This is called the recursive leap of faith. You verify:

  1. The base case is correct (factorial(1) should be 1 — yes)
  2. The recursive call moves toward the base case (n-1 is smaller — yes)
  3. Assuming the recursive call works, the combination is correct (n × (n-1)! = n! — yes)

If all three hold, the method works. You don't need to trace every level.

Recursive Count-Up

What if you print after the recursive call? 

public static void countup(int n) {
    if (n <= 0) return;       // BASE
    countup(n - 1);           // recurse first
    System.out.print(n + " "); // print on the way BACK
}

countup(3) prints: 1 2 3

The printing happens during the unwinding — as each call returns, it prints its n. The deepest call (countup(0)) returns first, then countup(1) prints 1, then countup(2) prints 2, then countup(3) prints 3.

Common Traps

Trap What happens Fix
No base case StackOverflowError Add stop condition that always gets reached
Recursive call moves away from base Same — never stops n → n-1 (toward 0). NOT n → n+1 (away)
Base case unreachable Same — e.g., step by 2, base is at 1, never hits it exactly Use n <= 1 not n == 1, or carefully design step size
Forgetting return value Wrong result (void methods excepted) Every non-void branch must return a value

Socratic Checkpoint

1. The Anatomy of Stack Overflow

We know that a loop like while (true) {} will run forever but will not crash the program due to a memory error, whereas a recursive method with no base case causes an immediate StackOverflowError and crashes. Explain this difference in terms of the JVM's call stack. What physical resource is recursion consuming with every call that a simple loop does not?

2. The Unwinding Echo

Compare these two recursive printing patterns:

// Count Down
public static void count1(int n) {
    if (n <= 0) return;
    System.out.print(n + " ");
    count1(n - 1);
}

// Count Up
public static void count2(int n) {
    if (n <= 0) return;
    count2(n - 1);
    System.out.print(n + " ");
}

Why does count1(3) print 3 2 1 while count2(3) prints 1 2 3? How does the JVM use stack frames to "remember" the values of n until the recursion starts unwinding?

Bug Hunt: The Odd Skipping Trap

A student writes a recursive method intended to sum all positive numbers from n down to 0, skipping by 2 (e.g., for even numbers: 6 + 4 + 2 = 12). However, when they test it with an odd integer like sumEvenSkip(3), the program hangs for a second and then crashes with a massive java.lang.StackOverflowError trace.

public static int sumEvenSkip(int n) {
    if (n == 0) {
        return 0;
    }
    return n + sumEvenSkip(n - 2); // Crashes if n is odd!
}

Question: Spot the bug! Why does an odd integer input like 3 bypass the base case entirely? How can you modify the base case condition to make this method safe for all positive integer inputs?

Tinker Challenge: Recursive Range Sum & Indented Trace

Goal: Write recursive accumulators and visualize the call stack dynamically.

  1. Range Accumulation: Write a method public static int sumRange(int start, int end) that recursively calculates the sum of all integers from start to end inclusive. For example, sumRange(3, 6) must return 18 (3 + 4 + 5 + 6).
    Safety First: Guard against invalid ranges! If the user inputs a range where start > end, your method must safely return 0 rather than descending into a stack overflow.
  2. Visual Stack Tracer (Boss Move): Modify your sumRange method (or create a helper) that accepts a depth level and prints out an indented trace of the calls. For example, calling sumRange(3, 5) should print: 
    -> sumRange(3, 5)
  -> sumRange(4, 5)
    -> sumRange(5, 5)
      [Base Case Hit: returning 5]
    [Unwinding: returning 5 + 4 = 9]
  [Unwinding: returning 9 + 3 = 12]
