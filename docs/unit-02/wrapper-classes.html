Wrapper Classes

Primitives (int, double, boolean) are fast and simple. But sometimes you need them to behave like objects. That's what wrapper classes do — they wrap a primitive value in an object.

The Wrappers

PrimitiveWrapper Class
intInteger
doubleDouble
booleanBoolean
charCharacter

Constants — Named Magic Numbers

int biggest = Integer.MAX_VALUE;   // 2147483647
int smallest = Integer.MIN_VALUE;  // -2147483648

double maxDouble = Double.MAX_VALUE; // huge number

// Much more readable than magic numbers:
if (score > Integer.MAX_VALUE - 100) { ... }

Parsing — String to Number

This is the most common use. User types a number... but Scanner already does this for you with nextInt(). So when would you use parsing directly? 

// When the number is embedded in data:
String line = "42,18,99";  // like reading from a file later
String[] parts = line.split(",");
int first = Integer.parseInt(parts[0]);  // "42" → 42

// Or validating input:
String userInput = "hello";
try {
    int n = Integer.parseInt(userInput); // throws exception — not a number
} catch (NumberFormatException e) {
    System.out.println("That's not a number!");
}

Autoboxing — Java Does It For You

// You rarely need to wrap/unwrap manually.
// Java automatically converts between primitive and wrapper:

Integer wrapped = 42;        // autoboxing: int → Integer
int unwrapped = wrapped;     // unboxing: Integer → int

// This also works:
ArrayList list = new ArrayList<>();
list.add(5);                 // autoboxing happens automatically

When To Use Wrappers vs Primitives

Use primitives (int, double): almost always. They're faster, simpler, and can't be null.

Use wrappers (Integer, Double):

  • When using ArrayList or other generic collections (Unit 7)
  • When you need the constants (MAX_VALUE, MIN_VALUE)
  • When a value might be "unknown" — primitives can't be null

Tinker Challenge: The Caching Trap

Integer a = 100;
Integer b = 100;
Integer c = 200;
Integer d = 200;

System.out.println(a == b);  // ???
System.out.println(c == d);  // ???
System.out.println(a.equals(b)); // ???
System.out.println(c.equals(d)); // ???

Run it. Wait, what happened? Why is a == b true but c == d is false?

Socratic Checkpoint
Java tries to save memory by caching small Integers (from -128 to 127). So a and b point to the exact same cached object in memory. But 200 is too big to be cached, so c and d point to different objects in memory, even though they hold the same value.
Knowing this, what method should you ALWAYS use to compare Integer objects, just like with Strings?

Bug Hunt: Unboxing Dangers

Bug 1: The Null Pointer
Integer score = null;
int finalScore = score;

Run this. It causes a NullPointerException. Why? What happens when Java tries to unbox a null wrapper into a primitive?

Bug 2: The Parsing Crash
String input = "10.5";
int value = Integer.parseInt(input);

Run this. It throws a NumberFormatException. Look closely at the string "10.5". Why can't `Integer.parseInt` handle it? How would you fix it?
