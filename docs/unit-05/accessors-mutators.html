Accessors and Mutators (Getters and Setters)

Instance variables are private. That means only the class itself can touch them directly. So how does the rest of your program read or change your object's data? Through methods.

Why Encapsulation?

Without encapsulation, anyone can corrupt your object:

// WITH public variables — dangerous!
player.hp = -999;       // negative HP? Impossible to kill.
player.ac = 99999;      // invincible. Cheating is trivial.

// WITH private variables + setters — safe!
player.setHP(-999);     // setter clamps to 0
player.setAC(99999);    // setter caps at reasonable max

Encapsulation means you write the rules. Your methods are the only way in. If someone wants to change your object, they play by your rules.

Accessors (Getters)

A getter returns a value. It reads data. It changes nothing. The naming convention: get + VariableName. 

public class Player {
    private String name;
    private int hp;

    public String getName() {   // returns the name
        return name;
    }

    public int getHP() {        // returns current HP
        return hp;
    }
}

// Usage:
Player p = new Player("Hero", 20, 14);
System.out.println(p.getName());  // "Hero"
if (p.getHP() < 10) { ... }       // check without touching private data

Mutators (Setters)

A setter changes a value — after checking that the new value is valid. Naming: set + VariableName. Usually void — they do things, they don't return things. 

public void setHP(int newHP) {
    if (newHP < 0) {
        hp = 0;              // floor: can't go below 0
    } else if (newHP > maxHP) {
        hp = maxHP;          // ceiling: can't exceed max
    } else {
        hp = newHP;
    }
}

Do I Always Need Both?

No. Some variables should be read-only (getter, no setter):

private final String id;     // ID never changes after creation
public String getID() { return id; }
// No setID() — there shouldn't be one.

Some should be internal only (no getter, no setter):

private int tempCalculation;  // used internally, nobody else needs it
// No getter. No setter. This variable is the class's business.

The rule: only expose what other code needs. Start with nothing public. Add getters and setters only when another class actually calls them.

The this Keyword with Setters

When the parameter has the same name as the instance variable, you need this to tell them apart: 

public void setName(String name) {
    this.name = name;    // this.name = instance var, name = parameter
}

Even when there's no shadowing, adding this. makes your intent clear. It's never wrong to use this. for instance variables.

Pattern: Combat Methods Through Setters

public void takeDamage(int amount) {
    if (amount < 1) amount = 1;   // minimum 1 damage
    setHP(hp - amount);            // use setHP for bounds checking
}

public void heal(int amount) {
    if (amount < 0) return;        // negative healing = do nothing
    setHP(hp + amount);            // setHP caps at maxHP
}

public boolean isAlive() {
    return hp > 0;
}

What might happen in an RPG if a boss's maximum health could be changed by any other part of the code mid-combat?

Bug Hunt: The Phantom Setter

A student wrote a setter to update their player's speed, but no matter what they do, their player always moves at the default speed of 10. Spot the missing keyword that is causing this "shadowing" bug, where the parameter just updates itself:

public class Player {
    private int speed = 10;

    public void setSpeed(int speed) {
        speed = speed;
    }
}

Tinker Challenge: The Fragile Blade

Write the getters and setters for a Sword class that manages its own durability using strict encapsulation.

  1. Variables: Provide a private durability (int, representing 0 to 100) and a private name (String).
  2. Getters: Write accessors for both variables.
  3. Setter Validation: Write a setDurability(int dur) method. If the new value is greater than 100, clamp the durability to 100. If the new value is less than 0, clamp it to 0. Otherwise, update it to the new value.
  4. Action Methods: Write a strike() method that lowers the sword's durability by 5 (by calling your own setDurability method to get the validation for free). Write a repair() method that increases it by 20 (again, use the setter!).
