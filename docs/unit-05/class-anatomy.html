Anatomy of a Class

A class is a blueprint. It defines what every object of that type looks like and can do. One class → many objects. Write it once, use it everywhere.

The Four Sections

Every class you write has these four parts:

public class ClassName {
    // 1. INSTANCE VARIABLES — the data (each object has its own copy)
    private type variableName;

    // 2. CONSTRUCTOR(S) — how to build an object (runs when you use new)
    public ClassName(parameters) {
        // set up initial values
    }

    // 3. ACCESSORS (GETTERS) — read the data safely
    public type getVariable() {
        return variable;
    }

    // 4. MUTATORS + METHODS — change the data, perform actions
    public void setVariable(type newValue) {
        // validate, then assign
    }

    public String toString() {
        // return a readable string representation
    }
}

Section 1: Instance Variables

Instance variables store the object's state. Every object gets its own copy. One Sword might have damage = 12, another damage = 45. Same blueprint. Different values. 

public class Sword {
    private String name;      // each sword has its own name
    private int damage;       // each sword has its own damage
    private int durability;   // each sword has its own durability
}
Always private. Instance variables are private. Other classes can't touch them directly. They have to use your methods. This is called encapsulation — you control how data changes.

Section 2: Constructor

The constructor runs when you write new. Its job: set up the object so it's ready to use. The constructor name always matches the class name. It has no return type (not even void). 

public Sword(String n, int dmg, int dur) {
    name = n;
    damage = dmg;
    durability = dur;
}

Section 3: Accessors (Getters)

Getters let other code read your object's data without touching the private variables directly. They return a value but change nothing. 

public String getName() {
    return name;
}

public int getDamage() {
    return damage;
}

Section 4: Mutators (Setters) and Methods

Setters let other code change your object's data — but only through your rules. This is where validation lives. 

public void setDurability(int dur) {
    if (dur < 0) {
        durability = 0;       // can't go below zero
    } else if (dur > 100) {
        durability = 100;     // capped at 100
    } else {
        durability = dur;
    }
}

Other methods do things: attack, defend, level up, take damage. They use the instance variables to perform actions.

Socratic Checkpoint: The Wall of Privacy

Look at the Sword class above. Why are variables like damage marked as private? What disaster could happen in our RPG if another programmer (or another part of your code) could write mySword.damage = -9999; directly from a different file?

Bug Hunt: The Broken Blueprint

A student tried to write a Shield class, but it has several critical flaws. Find the three structural errors in their blueprint:

public class Shield {
    String name;
    int defense;

    public void Shield(String n, int def) {
        name = n;
        defense = def;
    }

    public int getDefense() {
        return def;
    }
}

Hint: Think about privacy rules, what constructors are allowed to return, and variable scope.

Tinker Challenge: The Potion Vendor

Write a complete Potion class from scratch containing all four structural sections.

  1. Instance Variables: name (String), healingAmount (int), isEmpty (boolean). Ensure they follow the rule of encapsulation!
  2. Constructor: Takes a name and a healing amount as parameters. Sets isEmpty to false automatically.
  3. Getters: Write accessors to return the name and the healing amount.
  4. Action Method: Write a drink() method. If isEmpty is false, print "You drank the [name]", set isEmpty to true, and return the healingAmount. If it is already empty, print "The bottle is empty" and return 0.
