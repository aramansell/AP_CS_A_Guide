Designing a Class from a Spec

So far, you've built classes following step-by-step instructions. But real software starts with a specification — a description of what the class should do. You decide: what variables? What methods? What's public? What's private?

The Design Process

  1. Identify the data (instance variables). What does this object need to know?
  2. Decide: mutable or immutable? Does the data change after creation? If not → final.
  3. Write the constructors. How should other code create objects of this type?
  4. Write the getters. What data does other code need to read?
  5. Write the methods. What actions does this object perform?
  6. Write toString(). How does this object describe itself?

Example: Item Class

The spec: Build an Item class for a fantasy RPG.

  • An item has: name, weight (1-50 lbs), gold value, and rarity (Common, Uncommon, Rare, Epic, Legendary)
  • You need to: create items, check if they can be carried (total weight matters), compare rarity, print descriptions
  • Items are immutable — once created, they don't change

Step 1: Identify data

  • name — String
  • weight — int (1-50)
  • goldValue — int
  • rarity — String ("Common", "Uncommon", "Rare", "Epic", "Legendary")

Step 2: Immutable → all final

private final String name;
private final int weight;
private final int goldValue;
private final String rarity;

Step 3: Constructor(s)

// Full constructor — weight clamped 1-50
public Item(String name, int weight, int value, String rarity) {
    this.name = name;
    this.weight = Math.max(1, Math.min(50, weight));
    this.goldValue = Math.max(0, value);
    this.rarity = rarity;
}

// Name + value only — defaults: weight=1, rarity="Common"
public Item(String name, int value) {
    this(name, 1, value, "Common");
}

Step 4-5: Getters + Methods

// Getters (no setters — immutable!)
public String getName() { return name; }
public int getWeight() { return weight; }
public int getGoldValue() { return goldValue; }
public String getRarity() { return rarity; }

// Methods
public boolean isHeavy() { return weight > 25; }

public boolean isRarerThan(Item other) {
    // Higher rarity index = rarer
    return rarityIndex() > other.rarityIndex();
}

private int rarityIndex() {
    if (rarity.equals("Legendary")) return 4;
    if (rarity.equals("Epic"))      return 3;
    if (rarity.equals("Rare"))      return 2;
    if (rarity.equals("Uncommon"))  return 1;
    return 0;  // Common
}

Step 6: toString()

public String toString() {
    return name + " [" + rarity + "] — " + goldValue + " gold, " + weight + " lbs";
}

public vs private: Design Heuristic

publicprivate
ConstructorsInstance variables
Getters (most)Helper methods (internal only)
Setters (some)Constants used internally
Main behavior methodsAnything that could be misused if public
toString()—

Start private. Make something public only when another class actually needs to call it. You can always open things up later. You can't take public back once code depends on it.

Socratic Checkpoint: The Immutable Rule

We declared the variables in the Item class as final, meaning they can never change once the constructor finishes. Why does this design make sense for an item like a "Health Potion" or "Bronze Sword"? When would you definitely NOT want to use final for your instance variables?

Bug Hunt: The Leaky Shield

A student tried to build an immutable Shield class, but their code failed the peer review. Find the two structural errors that ruin the immutability of this class:

public class Shield {
    private String name;
    private int defense;

    public Shield(String name, int defense) {
        this.name = name;
        this.defense = defense;
    }

    public void setDefense(int newDefense) {
        this.defense = newDefense;
    }
}

Tinker Challenge: The Spellbook

Design and build a Spell class from scratch following this exact spec:

  1. Variables: name (String), damage (int, 1-50), manaCost (int, 1-100). The spell's properties should be completely immutable!
  2. Constructors: Write a full constructor with validation (enforcing the damage and mana ranges above). Write a second constructor that takes only name and damage, defaulting manaCost to 10 (use constructor chaining).
  3. Getters: Write getters for all three variables. Do not write setters.
  4. Methods: Write a method public boolean canCast(int currentMana) that returns true if the player has enough mana to cast the spell, and false otherwise.
  5. toString: Make it describe the spell nicely, like this: "Fireball [Cost: 10] - 25 dmg"
