Student RPG Guide: Building Player.java

Since Unit 2, you've been typing Player p = new Player("Hero", 14, 12); without knowing what's inside. Now you write the file that makes it all work. No blanks. No starter code. Just the spec. You figure out the implementation.

What You're Building

The Player class — the data structure that holds your character's name, stats, and HP. Methods to take damage, heal, and report status. This replaces all those scattered int playerHP variables from previous units. One file: Player.java.

References

You've already built these classes with your teacher. They show you how — now you apply the same patterns to your own class:

Spec

Instance Variables

All private. None are static.

VariableTypePurpose
nameStringCharacter name, e.g. "Hero"
hitPointsintCurrent health
maxHitPointsintMaximum health (set at creation, never changes)
strengthintPhysical power
dexterityintAgility and reflexes
intelligenceintMental acuity
constitutionintToughness and endurance

Constructor

  • Parameters: String name, int hp, int str, int dex, int intel, int con
  • Store each parameter into its instance variable using this
  • maxHitPoints gets the same value as hp

Getters

One getter per instance variable. Standard pattern from Card and BankAccount.

HP Management

These are mutators — they change hitPoints. Use the guard-clause pattern from BankAccount's withdraw().

MethodWhat It Does
setHP(int newHP) Sets hitPoints to newHP.
Clamp: if newHP < 0, set to 0. If newHP > maxHitPoints, set to maxHitPoints.
takeDamage(int amount) Reduces HP by amount. Delegate to setHP() — don't modify hitPoints directly.
Print: "<name> takes <amount> damage! HP: <current>/<max>"
heal(int amount) Increases HP by amount. Delegate to setHP().
Print: "<name> heals <amount> HP! HP: <current>/<max>"
isAlive() Returns true if hitPoints > 0, false otherwise.

toString()

Returns a single-line string representation. When you call System.out.println(player), Java calls toString() automatically.

Format:

"<name> — HP: <current>/<max>, STR:<str> DEX:<dex> INT:<int> CON:<con>"

Test It

Write a main() in a separate runner class (or in Player.java itself). Verify every behavior:

// === Test 1: Constructor + toString ===
Player hero = new Player("Hero", 25, 14, 12, 16, 10);
System.out.println(hero);
// Expected: "Hero — HP: 25/25, STR:14 DEX:12 INT:16 CON:10"

// === Test 2: Getters ===
System.out.println(hero.getName());        // "Hero"
System.out.println(hero.getHitPoints());    // 25
System.out.println(hero.getMaxHitPoints()); // 25
System.out.println(hero.getStrength());     // 14

// === Test 3: takeDamage (normal) ===
hero.takeDamage(8);
System.out.println(hero.getHitPoints());    // 17
// Expected output: "Hero takes 8 damage! HP: 17/25"

// === Test 4: takeDamage (clamp to 0) ===
hero.takeDamage(100);
System.out.println(hero.getHitPoints());    // 0 — clamped!
System.out.println(hero.isAlive());         // false

// === Test 5: heal (normal) ===
hero.heal(15);
System.out.println(hero.getHitPoints());    // 15

// === Test 6: heal (clamp to max) ===
hero.heal(500);
System.out.println(hero.getHitPoints());    // 25 — clamped to max!
System.out.println(hero.isAlive());         // true

// === Test 7: isAlive boundary ===
Player dead = new Player("Ghost", 0, 10, 10, 10, 10);
System.out.println(dead.isAlive());         // false
dead.heal(1);
System.out.println(dead.isAlive());         // true

Socratic Checkpoints

1. Why does takeDamage() clamp HP to 0 instead of letting it go negative? What would break in your dungeon loop if HP could be negative?

2. You have two ways to check if a player is alive: hp > 0 and player.isAlive(). If someone later changes the HP logic (maybe a player is dead at -10 HP now means something), which approach would break less code? Why?
