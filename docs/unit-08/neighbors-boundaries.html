Neighbors and Boundary Safety

You've learned to traverse 2D arrays. Now the real challenge: checking the cells around a cell without crashing into the array's walls.

The Problem: ArrayIndexOutOfBoundsException

char[][] map = {
    {'#', '#', '#'},
    {'#', 'M', '#'},
    {'#', '#', '#'}
};

// Trying to check above the monster at (1,1):
char above = map[0][1];  // OK — row 0 exists

// But what if the monster is at (0,0)?
char above = map[-1][0];  // CRASH — row -1 doesn't exist
// OR at (2,2)?
char below = map[3][2];   // CRASH — row 3 doesn't exist (max row is 2)

The rule: Every time you access a neighbor, you MUST check bounds first. No exceptions. The compiler won't stop you, but the runtime will.

The Four Directions

From any cell at (row, col), its four orthogonal neighbors are:

DirectionCoordinatesBounds Check
Up(row-1, col)row > 0
Down(row+1, col)row < grid.length - 1
Left(row, col-1)col > 0
Right(row, col+1)col < grid[0].length - 1

The Safe Neighbor Pattern

The standard pattern wraps each access in its own bounds check:

public static int countMonsterNeighbors(char[][] grid, int row, int col) {
    int count = 0;

    // Up
    if (row > 0 && grid[row-1][col] == 'M') {
        count++;
    }
    // Down
    if (row < grid.length - 1 && grid[row+1][col] == 'M') {
        count++;
    }
    // Left
    if (col > 0 && grid[row][col-1] == 'M') {
        count++;
    }
    // Right
    if (col < grid[0].length - 1 && grid[row][col+1] == 'M') {
        count++;
    }

    return count;
}
Notice: Java uses short-circuit evaluation — if the first condition in an && is false, it never evaluates the second. That's why we put the bounds check FIRST: if it fails, we skip the array access entirely.

Compact Direction Arrays (Advanced)

Instead of four separate if-blocks, store the four direction offsets:

public static boolean hasTreasureNearby(char[][] grid, int row, int col) {
    // Offsets for: up, down, left, right
    int[] dRow = {-1, 1, 0, 0};
    int[] dCol = {0, 0, -1, 1};

    for (int d = 0; d < 4; d++) {
        int newRow = row + dRow[d];
        int newCol = col + dCol[d];

        // Bounds check using length, not hardcoded numbers
        if (newRow >= 0 && newRow < grid.length &&
            newCol >= 0 && newCol < grid[0].length) {

            if (grid[newRow][newCol] == '$') {
                return true;
            }
        }
    }
    return false;
}

This pattern scales: add diagonal offsets to go from 4 to 8 directions.

The #1 Trap: The Four-Branch Duplication Problem

// DANGEROUS — duplicated logic in every branch:
if (row > 0) {
    if (grid[row-1][col] == target) {
        System.out.println("Found at " + (row-1) + ", " + col);
        count++;
    }
}
// ... repeat for down, left, right ...

// If you need to CHANGE the action (print, count, return...),
// you must change it in FOUR places. Easy to miss one.

Solution: Extract the action into a helper method, OR use the direction-array pattern. One loop, one action, four directions handled cleanly.

Edge Cases: Wrapping vs. Clamping

When the player goes off the edge of the map, you have two choices: 

// CHOICE 1: Clamp — stop at the edge
int clampedRow = Math.max(0, Math.min(grid.length - 1, row + dRow));
int clampedCol = Math.max(0, Math.min(grid[0].length - 1, col + dCol));

// CHOICE 2: Wrap — appear on the opposite side (Pac-Man style)
int wrappedRow = (row + dRow + grid.length) % grid.length;
int wrappedCol = (col + dCol + grid[0].length) % grid[0].length;

Most RPG maps use clamping. Pac-Man uses wrapping. Know which one your game needs.

The Diagonal Question

Orthogonal neighbors (up, down, left, right) are sometimes called "4-directional." If you need diagonals too (8-directional), add these offsets: 

int[] dRow = {-1, -1, -1,  0, 0,  1, 1, 1};
int[] dCol = {-1,  0,  1, -1, 1, -1, 0, 1};

Now you can check all 8 surrounding cells with the same loop. Same logic, more directions.

Do edge cells have an advantage or disadvantage in this count? Why might you get a different answer if you treated off-the-edge cells as empty vs. if you treated them as walls?

Bug Hunt: The Over-Counting Monster Detector

A student writes code to count how many cells have at least one monster neighbor. They use a boolean to mark each cell once, but the output count is consistently wrong — sometimes too high, sometimes too low.

public static int cellsWithMonsterNeighbor(char[][] grid) {
    int count = 0;
    for (int r = 0; r < grid.length; r++) {
        for (int c = 0; c < grid[0].length; c++) {
            // Check all four neighbors
            if (r > 0 && grid[r-1][c] == 'M') count++;
            if (r < grid.length-1 && grid[r+1][c] == 'M') count++;
            if (c > 0 && grid[r][c-1] == 'M') count++;
            if (c < grid[0].length-1 && grid[r][c+1] == 'M') count++;
        }
    }
    return count;
}

Question: A single monster cell at (1,1) with empty cells all around it will increment count multiple times. Why? Trace which cells trigger the count increment and explain why the answer is almost certainly too high. How would you fix it?

Tinker Challenge: Flood Zone Calculator

Goal: Build a method that analyzes every cell's surroundings in one pass.

  1. Danger Level Map: Write a method public static int[][] dangerGrid(char[][] map) that returns a new 2D int array where each cell contains the number of monster ('M') neighbors in its four orthogonal directions. Edge cells should not crash.
  2. Safe Zone Finder: Write public static int[] findSafeZone(char[][] map) that locates the first cell with zero monster neighbors and returns its coordinates as {row, col}. If no safe zone exists, return {-1, -1}.
  3. Boss Move: Write public static boolean isSurrounded(char[][] map, int row, int col) that returns true if ALL valid orthogonal neighbors of the cell are monsters ('M'). A cell on the edge can never be fully surrounded (it has fewer neighbors). A cell in a corner has only two neighbors — do you consider it "surrounded" if both are monsters?
