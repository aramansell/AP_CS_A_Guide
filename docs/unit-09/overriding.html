Method Overriding

Inheritance gives the child everything the parent has. But sometimes "everything" isn't what you want. A Warrior's attack() should be different from a Mage's attack(). Overriding is how a child class says: "I'll handle this my own way."

The Problem Inheritance Creates

public class Character {
    public void attack() {
        System.out.println(name + " attacks!");
    }
}

public class Player extends Character {
    // No attack() here — inherits Character's version
}

// In main:
Player hero = new Player("Aria");
hero.attack();   // "Aria attacks!"  ... that's boring

Every player says the exact same thing. A warrior with an axe, a mage with magic, a rogue with daggers — all just say "attacks!" That's what overriding fixes.

@Override — Replace the Parent's Version

public class Warrior extends Player {
    @Override                         // ← ALWAYS include this
    public void attack(Character target) {
        int damage = getStrength() / 3;
        System.out.println(name + " swings their axe for " + damage + " damage!");
        target.takeDamage(damage);
    }
}

Now when a Warrior attacks:

Warrior thorn = new Warrior("Thorn");
thorn.attack(goblin);
// Output: Thorn swings their axe for 5 damage!

Three Rules of Overriding

  1. Same signature: Same method name, same parameter types, same return type. Exact match.
  2. Use @Override: The annotation tells Java "I intend to override." If you get the signature wrong, the compiler catches it.
  3. Access can't be more restrictive: If the parent method is public, the override must be public too.

Why @Override Matters

Without @Override, a typo becomes a silent new method:

// BUG — forgot @Override, misspelled the method name
public void atack(Character target) {   // atack ≠ attack
    // This is a NEW method, not an override!
    // The inherited attack() still runs.
    // No compiler error to warn you.
}

// FIXED — @Override catches the mistake
@Override
public void atack(Character target) {   // COMPILE ERROR: method doesn't override
    // ...
}
Always. Use. @Override. It's your safety net. Two characters of typing saves you from debugging a method that never gets called.

super.method() — Extend, Don't Replace

Sometimes you want the parent's version PLUS your own additions. Use super.method() to call the parent's version from inside your override: 

@Override
public String toString() {
    // Parent gives us: "Thorn [85/120]"
    String parentInfo = super.toString();
    // Add our own stuff
    return parentInfo + " | Warrior | STA:" + getStamina();
}
// Result: "Thorn [85/120] | Warrior | STA:20"

This is the "extend, don't replace" pattern. Call super.method() to get the parent's work, then add yours on top.

Override vs Overload

These two words sound similar but mean completely different things:

OverrideOverload
Where? Child class replaces parent's version Same class, multiple versions
Signature EXACT match (same name, same params, same return) Different parameters (count, type, or order)
Annotation @Override None
Which runs? Actual object type at runtime (polymorphism) Argument types at compile time 
// OVERLOAD — same class, different parameters
public void attack(Character target) { }       // version 1
public void attack(int x, int y) { }           // version 2 — OVERLOADED

// OVERRIDE — child replaces parent
@Override
public void attack(Character target) { }       // OVERRIDES Character's attack()

Real RPG Example: Different Attacks

// Warrior: big hits, strength-based
@Override
public void attack(Character target) {
    int damage = getStrength() / 3;
    System.out.println(name + " swings their axe for " + damage + " damage!");
    target.takeDamage(damage);
}

// Mage: spells, intelligence-based
@Override
public void attack(Character target) {
    int damage = getIntelligence() / 3;
    System.out.println(name + " casts a magic bolt for " + damage + " arcane damage!");
    target.takeDamage(damage);
}

// Rogue: precision, dexterity-based, chance to miss
@Override
public void attack(Character target) {
    int roll = (int)(Math.random() * 20) + 1;
    if (roll >= 10) {
        int damage = getDexterity() / 4;
        System.out.println(name + " strikes for " + damage + " damage! (roll: " + roll + ")");
        target.takeDamage(damage);
    } else {
        System.out.println(name + " misses! (roll: " + roll + ")");
    }
}

Same method signature. Three completely different behaviors. That's the power of overriding.

Common Traps

What you didResultFix
Forgot @Override Silent overload instead of override. Wrong method runs. Add @Override. If it doesn't compile, your signature is wrong.
Different parameter type Overload, not override. Parent's version still calls on superclass refs. Match the parameter types EXACTLY.
Different return type Compile error (unless return is a subtype — "covariant return") Match the return type.
private override of public parent Compile error: "attempting to assign weaker access privileges" Match or widen the access modifier.

Socratic Checkpoint

1. The Timing of Decisions

We know that Method Overloading and Method Overriding represent two completely different kinds of polymorphism. Overloading is resolved at compile-time, while Overriding is resolved at runtime. In your own words, explain how the compiler makes decisions about overloaded methods versus how the JVM makes decisions about overridden methods. Why can't the compiler resolve overridden methods in advance?

2. The LSP (Liskov Substitution Principle) Access Restriction

Java strictly forbids a child class from "weakening" the access privileges of an overridden method (for instance, trying to override a public parent method as private in the child). Why is this limitation structurally mandatory for polymorphism to work? What catastrophe would occur if a developer held a child object inside a parent variable (e.g., Character c = new Thief();) and tried to call a method that was public on the parent but private on the child?

Bug Hunt: The Silent Overload

An RPG designer attempts to implement a custom Boss class that overrides the takeDamage method to apply a 50% damage reduction. However, when playtesting, they notice that calling takeDamage(12.5) on a Boss instance completely bypasses the damage reduction, dealing full damage!

public class Monster {
    public void takeDamage(double amount) {
        // Base damage logic
    }
}

public class Boss extends Monster {
    // Intended to override and reduce damage by half
    public void takeDamage(int amount) {
        System.out.println("Boss shrugs off some pain!");
        super.takeDamage(amount / 2.0);
    }
}

Question: Spot the bug! Why did the compiler allow this code to run without error, and why does a double damage value bypass the boss's custom reduction logic entirely? How would adding the @Override annotation above the boss's method have saved the developer?

Tinker Challenge: Berserker Override & Safe Extension

Goal: Safely extend parent behaviors through controlled overrides.

  1. Conditional super Call: Create a class Berserker that extends Player. Override the takeDamage(int amount) method. If the berserker's current HP is below 50, double the incoming damage to represent their high-risk rage state, and print a warning. Otherwise, keep damage normal.
    Safety First: Never copy-paste the parent's core logic inside the child override! Always route final calculations through super.takeDamage(finalAmount) so that general bookkeeping (like death detection or status effect tracking) is never bypassed.
  2. Override Verification: Write a small driver class where you create a Berserker with 100 HP, deal 20 damage (normal behavior), then deal 40 damage (which drops HP below 50), and then deal 20 more damage to verify that the damage doubling triggers dynamically!
