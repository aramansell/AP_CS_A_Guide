Object, Interfaces, and Casting

Inheritance gets you far. But three concepts sit at the top of the Java type system and unlock the rest of the language: the Object class (every class's secret parent), interfaces (contracts without inheritance), and casting (telling the compiler "trust me, this is actually a _____").

The Object Class

Every class in Java extends Object. If you write class Player { }, Java secretly makes it class Player extends Object { }. You don't need to type it.

Why does this matter? Every object inherits methods from Object:

MethodWhat it does
toString()Returns a String representation. Default: ClassName@hexAddress. Override it.
equals(Object obj)Tests if two objects are "logically equal." Default: == (same memory address). Override it.
hashCode()Returns an int hash for HashMap/HashSet. Must be consistent with equals().
getClass()Returns the runtime class. Used with reflection (rare in AP).

Overriding toString()

public class Player {
    private String name;
    private int hp;

    public Player(String name, int hp) {
        this.name = name;
        this.hp = hp;
    }

    @Override
    public String toString() {
        return name + " [" + hp + " HP]";
    }
}

// Usage:
Player p = new Player("Aria", 100);
System.out.println(p);  // Before override: Player@1a2b3c4d
                        // After override:  Aria [100 HP]
The @Override annotation: Not required, but always use it. If you accidentally mistype the method name (toSring), the compiler catches it because no parent has that method. Without @Override, you silently write a new method that never gets called.

Overriding equals()

public class Player {
    private String name;
    private int idNumber;

    @Override
    public boolean equals(Object obj) {
        // 1. Same reference? Always equal
        if (this == obj) return true;

        // 2. Null or wrong class? Never equal
        if (obj == null || getClass() != obj.getClass()) return false;

        // 3. Cast and compare fields
        Player other = (Player) obj;
        return idNumber == other.idNumber;
    }
}

// Usage:
Player p1 = new Player("Aria", 1001);
Player p2 = new Player("Aria", 1001);
System.out.println(p1 == p2);       // false — different objects
System.out.println(p1.equals(p2));  // true — same ID

The pattern: accept Object as the parameter (it MUST be Object, not Player), check for same-reference and null/wrong-class, then cast and compare your own fields. Every equals() method follows this pattern.

Casting: Telling the Compiler What You Know

When you hold a parent-type reference to a child object, the compiler only sees the parent's methods. Casting lets you access the child's specific methods. 

Character c = new Warrior("Thorn", 150, 15);  // Character reference, Warrior object

c.attack();   // OK — attack() is in Character
// c.rage();  // ERROR — the compiler sees Character, not Warrior

// Downcast to access Warrior methods:
Warrior w = (Warrior) c;
w.rage();     // OK — w is a Warrior reference

The instanceof Guard

Never cast without checking first. A wrong cast throws ClassCastException at runtime.

public static void useSpecial(Character c) {
    if (c instanceof Warrior) {
        Warrior w = (Warrior) c;
        w.rage();
    } else if (c instanceof Mage) {
        Mage m = (Mage) c;
        m.castSpell();
    }
}

// Wrong — no guard:
Warrior w = (Warrior) c;  // If c is actually a Mage, CRASH
instanceof checks the RUNTIME type, not the reference type. Character c = new Warrior("Thorn", 150, 15); c instanceof Warrior → true (the actual object IS a Warrior) c instanceof Character → true (Warrior extends Character) c instanceof Mage → false (a Warrior is not a Mage)

Interfaces: Contracts Without Inheritance

An abstract class says "I am a _____." An interface says "I can do _____." You can only extend one class, but you can implement many interfaces. 

// Interface: a contract — any class that implements this MUST have these methods
public interface Attackable {
    void takeDamage(int damage);
    boolean isAlive();
}

public interface Lootable {
    Item dropLoot();
    int getGoldValue();
}

// A class can implement multiple interfaces:
public class Dragon implements Attackable, Lootable {
    private int hp;
    private int treasure;

    public void takeDamage(int damage) {
        hp -= damage;
    }

    public boolean isAlive() {
        return hp > 0;
    }

    public Item dropLoot() {
        return new Item("Dragon Scale", treasure);
    }

    public int getGoldValue() {
        return treasure;
    }
}

Key difference from abstract classes: interfaces have NO instance variables, NO constructors, and (before Java 8) NO method bodies. They are pure contracts.

Polymorphism Through Interfaces

// A method that works with ANY Attackable:
public static int attackAll(Attackable[] targets, int damage) {
    int survivors = 0;
    for (Attackable t : targets) {
        t.takeDamage(damage);
        if (t.isAlive()) survivors++;
    }
    return survivors;
}

// Works on players, monsters, dragons, doors, anything Attackable:
Attackable[] targets = {player, dragon, boss, door};

This is the same polymorphic pattern you learned with abstract classes. Interfaces give you the same power without forcing a shared parent class.

Interfaces vs. Abstract Classes

FeatureAbstract ClassInterface
Instance variablesYesNo (only constants)
ConstructorsYesNo
Method bodiesYes (partial implementations)No* (pure contracts)
How many can you have?One parentAs many as you want
Relationship"is-a" (Warrior IS a Character)"can-do" (Dragon CAN be attacked)

*Java 8+ allows default methods in interfaces, but that's beyond AP scope.

Socratic Checkpoint

1. The Broken equals()

A student writes this equals method:

public boolean equals(Player other) {
    return this.id == other.id;
}

It works perfectly when they test it with p1.equals(p2). But later, when they put Players in an ArrayList and call list.contains(somePlayer), it never finds a match. Why? What's the difference between equals(Player) and equals(Object)?

2. Interface Blindness

You have a variable Attackable a = new Dragon("Smaug", 500, 1000);. What happens if you try to call a.getGoldValue()? Why does the compiler reject it even though Dragon has the method? What's the fix?

Bug Hunt: The Silent instanceof Trap

A game has three character types: Warrior, Mage, and Rogue. The combat system uses instanceof to apply type-specific effects. But the Rogue never gets its backstab bonus.

public static void applyTurnEffects(Character c) {
    if (c instanceof Warrior) {
        ((Warrior) c).berserk();
    } else if (c instanceof Mage) {
        ((Mage) c).rechargeMana();
    }
    // Rogues were supposed to get a backstab here...
}

Bug 1: Where's the Rogue? What happens when a Rogue enters combat — do they silently skip, or get treated as something else?

Bug 2: A new developer copies this pattern into a new method but forgets to update the instanceof chain. The compiler doesn't warn them. Why is instanceof-based dispatch dangerous in a growing codebase? What alternative pattern would the compiler force you to handle every type?

Tinker Challenge: Combat System with Interfaces

Goal: Refactor type-specific behavior into interfaces.

  1. Design an Interface: Create a SpecialAbility interface with one method: void useSpecial(). Have Warrior, Mage, and Rogue each implement it differently (rage, cast, backstab).
  2. Eliminate instanceof: Rewrite applyTurnEffects to call c.useSpecial() directly through the interface. No casting, no instanceof. Why is this safer?
  3. Boss Move: Create an Item interface with methods String getName() and void use(Character target). Make Potion, Scroll, and Weapon implement it. Store them all in an Item[] and let the player invoke use() on any item without knowing its type.
