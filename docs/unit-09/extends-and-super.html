extends and super

Inheritance lets one class gain everything another class has — fields, methods, everything. The parent gives. The child receives, then adds its own stuff.

The IS-A Rule

Only use extends when the child truly IS-A parent:

  • A Warrior IS-A Player — yes, extend Player
  • A Goblin IS-A Monster — yes, extend Monster
  • A Player IS-A Monster — no! Don't do this
  • A Sword IS-A Player — no! Don't do this
Test: Say the sentence aloud: "A [Child] IS-A [Parent]." If it sounds true, inheritance works. If it sounds wrong, find a different relationship.

Basic Syntax

// Parent class — the giver
public class Character {
    protected String name;
    protected int hp;

    public Character(String name, int hp) {
        this.name = name;
        this.hp = hp;
    }

    public String getName() { return name; }
    public int getHp() { return hp; }

    public void takeDamage(int damage) {
        hp -= damage;
        if (hp <= 0) {
            System.out.println(name + " is defeated!");
        }
    }
}

// Child class — the receiver
public class Player extends Character {
    private int gold;

    public Player(String name, int hp) {
        super(name, hp);       // <-- call parent constructor
        this.gold = 0;
    }

    public int getGold() { return gold; }
}

After writing Player extends Character, a Player object has:

  • name and hp — inherited from Character
  • getName(), getHp(), takeDamage() — inherited from Character
  • gold and getGold() — Player's own additions

super() — Constructor Chaining

Every child constructor MUST call a parent constructor. super(args) does this. And it must be the first line — no exceptions. 

public class Monster extends Character {
    private int armorClass;

    public Monster(String name, int hp, int ac) {
        super(name, hp);          // ALWAYS first — calls Character(String, int)
        this.armorClass = ac;     // THEN do Monster-specific setup
    }
}
super() formCalls
super()Parent's no-argument constructor
super(name, hp)Parent's two-argument constructor
super(name, hp, 12)Parent's three-argument constructor
If you don't write super(): Java inserts super() for you. But that only works if the parent HAS a no-argument constructor. If the parent requires arguments (like our Character class), you'll get a compile error until you write an explicit super(name, hp).

The Constructor Chain

Every constructor call ripples up the tree:

new Warrior("Thorn")
    → Warrior("Thorn") calls super("Thorn", 120, 16, 10, 8)
        → Player(...) calls super("Thorn", 120)
            → Character("Thorn", 120) calls super()
                → Object() — the root of ALL Java classes

Object is the ultimate parent. Every class in Java ultimately extends Object. That's why even an empty class has toString(), equals(), and hashCode() — they're inherited from Object.

Access Modifiers: private vs protected vs public

public class Character {
    private String secretName;     // ONLY Character can touch this
    protected int hp;              // Character + children + same package
    public int getLevel() { }      // EVERYONE can touch this
}

When to use protected: Use it for fields that subclasses genuinely need direct access to. For everything else, use private with getters/setters. Protected is a deliberate choice — it says "this field is part of the family's shared knowledge."

What Children CAN'T Inherit

  • Private fields and methods — they're invisible to children. Only accessible through public/protected getters.
  • Constructors — constructors are NOT inherited. Each class writes its own. (But they must chain up with super().)

Common Traps

What you wroteWhat happensFix
Forgot super(args) Compile error: "no suitable constructor found" Add super(name, hp) as first line
super() not first Compile error: "call to super must be first" Move it to line 1
Tried to extend two classes Compile error — Java has single inheritance A class can only extend ONE parent
Accessing name from child when it's private Compile error: "name has private access" Use getName() or change to protected

Socratic Checkpoint

1. The Invisible Compiler Hand

Suppose you have a class GameEntity with only one constructor: public GameEntity(String id). If you write a subclass NPC extends GameEntity and define an empty constructor public NPC() {}, it will refuse to compile. Why does this compile error happen even though the constructor is completely empty? What is Java implicitly injecting, and how do you resolve it?

2. The Lifecycle of an Object

Why does Java structurally mandate that super() must be the absolute first statement in a constructor? If Java allowed subclass initialization or method calls before the parent constructor executed, what would happen if the subclass constructor tried to read an inherited field that had not yet been set up by the parent constructor?

Bug Hunt: The Premature Setup

A designer creates a custom Rogue class extending our Player class, intending to initialize a custom stealthLevel. However, the Java compiler raises a red flag: call to super must be first statement in constructor.

public class Rogue extends Player {
    private int stealthLevel;

    public Rogue(String name, int hp, int stealthLevel) {
        this.stealthLevel = stealthLevel;
        super(name, hp); // Compilation fails here!
    }
}

Question: Identify why the compiler rejects this arrangement. How does initializing this.stealthLevel before super() violate Java's constructor rules? Correct the code so it compiles cleanly.
