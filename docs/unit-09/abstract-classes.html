Abstract Classes

A Character should never exist on its own. You should never create a generic "Character" object. A character is ALWAYS a Player or a Monster — something specific. Abstract classes enforce this rule at the compiler level.

The Problem: Vague Objects

Character someone = new Character("???", 50);
someone.attack(enemy);
// What does this even mean? A generic blob attacks?
// This makes no sense in our game world.

new Character() creates a vague, meaningless object. We want to FORBID it. Solution: make Character abstract.

Making a Class Abstract

public abstract class Character {      // ← abstract keyword
    protected String name;
    protected int hp;
    protected int maxHp;
    protected boolean alive;

    public Character(String name, int hp) {
        this.name = name;
        this.hp = hp;
        this.maxHp = hp;
        this.alive = true;
    }

    public abstract void attack(Character target);   // ← abstract method — no body!
}

What Abstract Changes

Concrete classAbstract class
Character c = new Player(...); ✓ Same — superclass refs still work
new Character("???", 50); — possible but bad design new Character("???", 50); — COMPILE ERROR. Can't instantiate.
Child classes CAN override methods Child classes MUST implement abstract methods (or be abstract themselves)
Methods have bodies Abstract methods have NO body — just a semicolon
Abstract = "I'm a category, not a thing." An abstract class exists to organize shared code and enforce what subclasses must do. It is intentionally incomplete.

Abstract Methods — Enforcing a Contract

An abstract method has NO body. It's a PROMISE that subclasses must keep: 

public abstract class Character {
    public abstract void attack(Character target);    // No { } — just ;
    public abstract String getBattleCry();             // Another contract
}

// CONCRETE subclass — MUST implement ALL abstract methods:
public class Warrior extends Player {
    @Override
    public void attack(Character target) {      // MUST have this
        // warrior-specific attack
    }

    @Override
    public String getBattleCry() {              // MUST have this
        return "For glory!";
    }
}

If you forget to implement an abstract method, the COMPILER catches it: "Warrior is not abstract and does not override abstract method getBattleCry()"

Abstract Classes Can Have Concrete Methods Too

Abstract doesn't mean "everything is abstract." Most methods can be concrete:

public abstract class Character {
    protected String name;
    protected int hp;

    public Character(String name, int hp) {     // Concrete constructor — children call with super()
        this.name = name;
        this.hp = hp;
    }

    public String getName() { return name; }    // Concrete method — works for all subclasses

    public void takeDamage(int damage) {        // Concrete method
        hp -= damage;
        if (hp <= 0) {
            System.out.println(name + " defeated!");
        }
    }

    public abstract void attack(Character target);  // Abstract — subclasses must provide this
}

Abstract = "I have SOME parts that are abstract." The concrete parts are inherited normally. Only the abstract parts must be provided by subclasses.

When to Use Abstract

ScenarioUse abstract?
The class represents a category, never a specific thing (Character, Vehicle, Shape) YES — make it abstract
You want to FORCE all subclasses to provide a method YES — make that method abstract
You want a default method that subclasses CAN override if they want NO — make it a concrete method. Subclasses can still override it.
The class is fully functional and should be instantiated directly (Goblin, Warrior) NO — leave it concrete

The Hierarchy in Our RPG

Character (abstract)
├── Player (abstract — you don't make a generic "Player," you pick a class)
│   ├── Warrior (concrete)
│   ├── Mage (concrete)
│   └── Rogue (concrete)
└── Monster (concrete? or abstract — depends on design)
    ├── Goblin (concrete)
    ├── Skeleton (concrete)
    └── Dragon (concrete)

Character is abstract because "new Character()" is meaningless. Player could also be abstract if you want to force hero class selection. The leaf classes — Warrior, Goblin, Dragon — are concrete because they are specific, playable things.

RPG Example: The Full Character Class

public abstract class Character {
    protected String name;
    protected int hp;
    protected int maxHp;
    protected boolean alive;

    public Character(String name, int hp) {
        this.name = name;
        this.hp = hp;
        this.maxHp = hp;
        this.alive = true;
    }

    // Concrete — works for everyone
    public String getName()  { return name; }
    public int getHp()       { return hp; }
    public int getMaxHp()    { return maxHp; }
    public boolean isAlive() { return alive; }

    // Concrete — shared behavior
    public void takeDamage(int damage) {
        hp -= damage;
        System.out.println(name + " takes " + damage + " damage. (" + hp + "/" + maxHp + ")");
        if (hp <= 0) {
            hp = 0;
            alive = false;
            System.out.println("  >>> " + name + " has been defeated!");
        }
    }

    // Concrete — children can override for custom display
    @Override
    public String toString() {
        return name + " [" + hp + "/" + maxHp + "]";
    }

    // Abstract — every subclass MUST define how it attacks
    public abstract void attack(Character target);
}

Common Traps

What you didResultFix
Tried new AbstractClass() Compile error: "abstract class cannot be instantiated" Use a concrete subclass: new Warrior(...)
Abstract method has a body { } Compile error: "abstract methods cannot have a body" Remove the body, end with ;
Concrete subclass missing abstract method Compile error: "not abstract and does not override..." Implement the method, or make the subclass abstract too
Made a class abstract that should be concrete Can't create instances you need Remove abstract — leaf classes should be concrete
Made EVERY method abstract No shared behavior inherited — what's the point? Use abstract only for methods that MUST differ per subclass

Socratic Checkpoint

1. The Partial Blueprint

We often call an abstract class a "partial blueprint." It provides some concrete implementations (like takeDamage()) but leaves others as abstract method stubs (like attack()). Why is this hybrid design so powerful? How does this force subclasses to align to a specific behavior contract while still letting them share standard code?

2. Abstract Classes vs. Interfaces

An abstract class is allowed to have instance variables (such as protected int hp;), whereas a Java interface is not allowed to have mutable instance state. In what scenario is an abstract class a much better architectural choice than a standard interface? How does having inherited fields change what a child class receives for free?

Bug Hunt: The Broken Contract

A programmer is building an NPC system. They write a concrete subclass NPC that extends their abstract GameCharacter class. However, the compiler throws a heavy red error: NPC is not abstract and does not override abstract method attack(Character) in GameCharacter.

public abstract class GameCharacter {
    public abstract void attack(Character target);
}

public class NPC extends GameCharacter {
    private String dialogue;

    public NPC(String name, String dialogue) {
        // Assume super setup works
        this.dialogue = dialogue;
    }

    public void talk() {
        System.out.println(dialogue);
    }
}

Question: Spot the issue! Why did the compiler fail to compile the NPC class? What are the two distinct options a developer has to resolve this specific compilation failure?

Tinker Challenge: The Peaceful Healer & Subclass Contracts

Goal: Implement and experiment with abstract methods under unique semantic constraints.

  1. Creative Contract Fulfillment: Create a new concrete subclass Healer that extends Character (or GameCharacter). The parent class forces you to override public void attack(Character target). However, Healers do not deal damage! Provide an implementation of attack that instead adds 15 HP to the target, representing a healing spell.
    Safety First: You cannot change the parent's signature to heal(Character) to make it fit your subclass style. A concrete class must implement all abstract methods with the exact signature defined by the parent, even if the internal behavior differs from the method's name!
  2. Verify the Healing "Attack": Write a short driver class. Create a Warrior with 50/100 HP and a Healer. Put both in a polymorphic array: Character[] battlefield = {warrior, healer};. Loop through the array, having each character call attack(warrior), and print the warrior's HP after each round to prove that the healer successfully healed them!
