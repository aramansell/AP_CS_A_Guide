Polymorphism

Polymorphism = "many forms." A Character variable can hold a Warrior, a Mage, a Goblin, or a Dragon. When you call attack() on that variable, the RIGHT version runs based on the actual object type — not the variable type. This is the single most powerful feature of inheritance.

The Big Idea

// Three different objects, one variable type:
Character c1 = new Warrior("Thorn");   // Character ref → Warrior object
Character c2 = new Mage("Elara");      // Character ref → Mage object
Character c3 = new Goblin();           // Character ref → Goblin object

// Same method call, different behavior:
c1.attack(enemy);   // Warrior swings axe
c2.attack(enemy);   // Mage casts magic bolt
c3.attack(hero);    // Goblin claws

This is dynamic binding. Java doesn't decide which attack() to run until runtime, when it can look at the actual object.

Superclass References

A variable declared as Character can reference ANY object whose class extends Character: 

Character someone;
someone = new Player("Aria", 100, 16, 10, 8);   // Player IS-A Character ✓
someone = new Monster("Goblin", 20, 10, "");     // Monster IS-A Character ✓
someone = new Warrior("Thorn");                   // Warrior IS-A Player IS-A Character ✓
someone = new String("hello");                    // COMPILE ERROR — String is NOT a Character

Compile-Time vs Runtime

This is the trickiest part of polymorphism. Two steps:

  1. Compile time: Can I call this method? → Look at the VARIABLE type.
  2. Runtime: Which version runs? → Look at the OBJECT type.
Character c = new Warrior("Thorn");

c.getName();        // Compile: getName() exists in Character? YES → compiles
                    // Runtime: c is actually a Warrior → runs Character version (or Warrior's if overridden)

c.powerStrike(x);   // Compile: powerStrike() exists in Character? NO → COMPILE ERROR
                    // Even though the object IS a Warrior and HAS powerStrike,
                    // the compiler only knows about the VARIABLE type (Character).
Memory aid: The compiler trusts the VARIABLE type. The JVM respects the OBJECT type. The compiler checks the door — the JVM decides which room you enter.

Polymorphic Arrays

The real power: treating different objects uniformly in loops.

// Mixed party — any Character subclass:
ArrayList<Character> party = new ArrayList<>();
party.add(new Warrior("Thorn"));
party.add(new Mage("Elara"));
party.add(new Rogue("Shade"));

// One loop handles them all:
for (Character hero : party) {
    hero.attack(randomEnemy);    // RIGHT version runs for each hero
    System.out.println(hero);     // RIGHT toString() for each class
}

No if-statements needed. No type checking. One loop, polymorphic dispatch handles everything.

instanceof and Casting

Sometimes you DO need to call a method that only exists on a specific subclass. That's what instanceof and casting are for: 

for (Character hero : party) {
    hero.attack(enemy);   // Works for all Characters

    // But special abilities are class-specific...
    if (hero instanceof Warrior) {
        Warrior w = (Warrior) hero;      // Safe cast — we checked first
        w.powerStrike(boss);             // Only Warrior has this
    } else if (hero instanceof Mage) {
        Mage m = (Mage) hero;
        m.fireball(allEnemies);          // Only Mage has this
    }
}
ExpressionMeaning
hero instanceof Warrior"Is the object that hero references actually a Warrior?" Returns true or false
(Warrior) hero"Treat this Character reference AS a Warrior reference" — casting
Danger: Casting without instanceof is a gamble: 
Character c = new Monster("Goblin", 10);
Player p = (Player) c;   // RUNTIME ERROR: ClassCastException
Always check with instanceof before casting unless you are absolutely certain of the type.

RPG Combat with Polymorphism

Polymorphism transforms combat from this:

if (turn == "warrior") {
    warrior.attack(goblin);
} else if (turn == "mage") {
    mage.attack(goblin);
} else if (turn == "rogue") {
    rogue.attack(goblin);
}

Into this:

ArrayList<Character> party = new ArrayList<>();
// ... fill party ...

for (Character hero : party) {
    if (hero.isAlive()) {
        hero.attack(target);    // One line. Always correct.
    }
}

And adding a new class — say, a Paladin — requires zero changes to the combat loop. Just write the Paladin class extending Player, override attack(), and the loop handles it automatically.

Why Polymorphism Matters

Without polymorphism, your code has:

  • Long if-else chains checking types
  • Duplicated loops for each subclass
  • Changes in every file when you add a new class

With polymorphism, your code:

  • Has short, clean loops
  • Treats all subclasses uniformly
  • Grows painlessly — add a class, inherit, override, done

This is the entire reason inheritance exists. Not to save typing — to make your code open for extension without modification.

Socratic Checkpoint

1. The Dual Personality of an Object

In the line of code Character hero = new Warrior("Arthur");, we see two different types. What is the reference type of hero? What is the object type of hero? Which of these two types is checked by the compiler to decide if you are allowed to call a method? Which of these two types is checked by the JVM at runtime to decide which version of the method actually runs?

2. The Cleric's Silent Inheritance

Suppose the parent class Character has a method public void speak(). The subclass Wizard overrides speak() with a custom chant, but the subclass Cleric does not write a speak() method at all. If you have a polymorphic array Character[] party containing both a Wizard and a Cleric, and you loop through and call party[i].speak() on both, what happens? How does the JVM trace up the inheritance tree at runtime to find the method for the Cleric?

Bug Hunt: The Blind Compiler

A student writes code to make a character roar. They are positive that their Monster class has a public void roar() method. However, their code refuses to compile, showing a frustrating cannot find symbol - method roar() error on the second line.

Character actor = new Monster("Orc", 80);
actor.roar(); // Compile error occurs here!

Question: Find the compile bug! Why can't the compiler find the roar() method on actor even though actor holds a Monster at runtime? How can you write a safe type guard and cast to make this compile and run successfully?

Tinker Challenge: Polymorphic Party Utilities & Cast Guards

Goal: Manage groups of distinct subclasses uniformly while safely querying unique behaviors.

  1. Polymorphic Action Dispatcher: Write a utility method public static void prepareForBattle(Character[] party). Loop through the party and:
    • If the character is an instance of a Wizard, downcast it and call their subclass-exclusive meditate() method.
    • If the character is an instance of a Warrior, downcast it and call their subclass-exclusive sharpenSword() method.
    • For any other general Character, call their standard rest() method.
    Safety First: Never downcast a superclass reference to a subclass type without placing an explicit instanceof check directly beforehand! A blind cast will inevitably throw a java.lang.ClassCastException at runtime if the object happens to be a different subclass.
  2. Polymorphic Driver Trace: Write a driver method that sets up an array of three mixed characters (a general Character, a Wizard, and a Warrior), passes it to prepareForBattle, and demonstrates that the correct subclass actions fire without any compile or runtime issues!
