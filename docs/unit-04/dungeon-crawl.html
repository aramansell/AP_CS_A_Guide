Student RPG Guide: Dungeon Crawl Extended

This is YOUR guide. After DungeonLoop, this guide adds multi-floor dungeons, trap rooms, escape mechanics, and treasure rooms — all driven by nested loops and deeper loop logic. Instead of code blanks, you are provided with Acceptance Criteria: if your code passes every test, it works.

What You're Building

A DungeonCrawl.java that chains together room exploration, combat encounters, treasure collection, escape mechanics, and a final boss. This combines everything from Unit 4: while loops (combat), for loops (rooms), nested loops (multi-floor), and loop analysis (break/continue decisions).

How It Connects to the Unit 4 Concepts

for-loops: Outer loop over floors, inner loop over rooms — nested.
nested-loops: Each floor has multiple rooms. Two for loops, one inside the other. Outer controls floor number, inner controls room number.
string-loops: Building and displaying the inventory list by looping through a comma-separated String.
loop-analysis: When does break fire? How many times does each loop run? Trace it before running.

FILE: DungeonCrawl.java

Create a new file called DungeonCrawl.java. Every part on this page belongs to this ONE file.

Part 1: generateRoom() — Random Encounter Engine

Recall DungeonLoop.java. You wrote an if/else chain for combat/treasure/trap rooms. Extract that pattern into a reusable method that takes parameters for scaling.

Requirements:

  • Method signature: 
    public static boolean generateRoom(int[] playerHP, int playerAC, int playerAtk,
                                     int[] gold, String[] inventory,
                                     int floor, int roomNumber)
  • Parameters use arrays so the method can modify HP, gold, and inventory in place (primitives can't be modified by a method)
  • Generates a random room type: 0 = combat, 1 = treasure, 2 = trap

Combat (type 0):

  • Pick a monster name from an array (Goblin, Skeleton, Orc, Dark Elf, Troll, Wraith) using (roomNumber + floor - 2) % names.length as the index
  • Scale monster stats: HP = 5 + (floor * roomNumber * 2), AC = 10 + floor + roomNumber, atk = floor + roomNumber, dmg die = 4 + floor + roomNumber
  • Use a while loop (same as DungeonLoop.runCombat): player attacks first, monster counterattacks if alive
  • Player attack: roll d20, add bonus. Natural 20 = critical (d8 × 2). Total >= AC = hit (d8). Otherwise miss.
  • Monster counterattack (only if monsterHP > 0): roll d20, add bonus. Total >= AC = hit (roll dmg die). Otherwise miss.
  • If player dies (playerHP[0] <= 0), return false. If victory, add gold (10 * floor * roomNumber) and return true.

Treasure (type 1):

  • Add gold: Dice.roll(10) * floor + roomNumber added to gold[0]
  • Pick a random item from an array and append it to inventory[0] followed by a comma
  • Return true

Trap (type 2):

  • Deal damage: Dice.roll(6) + floor subtracted from playerHP[0]
  • If playerHP[0] <= 0 after damage, return false. Otherwise return true.

Socratic Checkpoint:

Why are playerHP, gold, and inventory passed as arrays instead of plain int and String? What's the difference between passing an array and passing a primitive?

Acceptance Test:

int[] hp = {20};
int[] gold = {0};
String[] inv = {""};
boolean alive = DungeonCrawl.generateRoom(hp, 14, 3, gold, inv, 1, 1);
System.out.println("Survived: " + alive + ", HP: " + hp[0] + ", Gold: " + gold[0]);
// Re-run several times to see all three room types

Part 2: exploreFloor() — Multi-Room Floor Loop

Recall for-loops. Each floor has a fixed number of rooms. Use a for loop. This is the inner loop of the nested structure.

Requirements:

  • Method signature: 
    public static boolean exploreFloor(int[] playerHP, int playerAC, int playerAtk,
                                     int[] gold, String[] inventory, int floor)
  • Print a floor header: "\n=== FLOOR [floor] ==="
  • A floor has 3 + floor rooms (later floors are longer)
  • For loop: room from 1 to roomsOnFloor (inclusive)
  • Each iteration calls generateRoom(). If it returns false, immediately return false (player died)
  • After all rooms: restore floor * 3 HP (rest at end of floor). Print the recovery. Return true.

Socratic Checkpoint:

Why is roomsOnFloor = 3 + floor instead of always 5? What does this mean for floor 3 vs floor 1?

Acceptance Test:

int[] hp = {25};
int[] gold = {0};
String[] inv = {""};
boolean survived = DungeonCrawl.exploreFloor(hp, 14, 3, gold, inv, 1);
System.out.println("Survived floor: " + survived);
// Verify room count matches 3 + floor = 4 rooms for floor 1
// Verify HP recovery after floor completion

Part 3: runDungeon() — Multi-Floor Outer Loop

Recall nested-loops. The dungeon is an outer for loop (floors) containing an inner for loop (rooms). Each iteration of the outer loop runs the entire inner loop.

Requirements:

  • Method signature: public static void runDungeon(int playerHP, int playerAC, int playerAtk, int totalFloors)
  • Print a big dungeon header
  • Wrap playerHP, gold (0), and inventory ("") in arrays so methods can modify them
  • Save starting HP for the summary
  • Outer for loop: floor from 1 to totalFloors (inclusive)
  • Each iteration calls exploreFloor(). If it returns false, print failure summary and return
  • After all floors: call bossFight(), then printSummary()

Socratic Checkpoint:

Draw the structure: an outer for loop contains exploreFloor, which contains a for loop, which calls generateRoom, which contains a while loop. Four levels of loops. Which loop controls what?

Acceptance Test:

DungeonCrawl.runDungeon(25, 14, 3, 3);
// Verify: 3 floors, 4+5+6 = 15 rooms max (fewer if player dies)
// Boss fight triggers after floor 3
// Summary prints at the end

Part 4: bossFight() — Final Encounter

Requirements:

  • Method signature is the same as Part 3 plus a floors param for scaling
  • Print a boss header
  • Boss name: "Ancient Dragon" if floors >= 3, "Ogre Lord" otherwise
  • Boss stats scale with floors: HP = 15 + floors * 10, AC = 14 + floors, atk = 4 + floors, dmg die = 8 + floors * 2
  • While loop (same combat pattern): player attacks first, then boss attacks if still alive
  • Boss CAN get critical hits (natural 20 = dmg die × 2)
  • If boss dies: print victory, add gold (floors * 50), add treasure to inventory, return
  • If player dies: print defeat

Socratic Checkpoint:

The boss fight uses the same while-loop combat pattern as generateRoom. Why not just call generateRoom with boss stats? What's different about the boss fight that warrants its own method?

Acceptance Test:

Run the full dungeon. Verify the boss fight triggers after all floors. Verify boss name depends on floor count. Verify boss can critical hit.

Part 5: printSummary() — End-of-Dungeon Report

Requirements:

  • Method signature: public static void printSummary(int finalHP, int startHP, int gold, String inventory)
  • Print HP remaining vs starting HP
  • Print total gold
  • Print inventory items: if inventory is empty, print "(empty)". Otherwise, split on "," and loop through each item, printing with " - " prefix

Acceptance Test:

DungeonCrawl.printSummary(10, 25, 75, "Health Potion,Iron Sword,Ancient Scroll,");
// Verify output:
// HP remaining: 10 / 25
// Gold collected: 75
// Inventory:
//   - Health Potion
//   - Iron Sword
//   - Ancient Scroll
DungeonCrawl.printSummary(0, 25, 30, "");
// Verify: Inventory: (empty)

Part 6: main() — Run It

Requirements:

  • Call runDungeon(25, 14, 3, 3) — 25 HP, 14 AC, +3 attack, 3 floors

Acceptance Test:

Run DungeonCrawl.main(). A full 3-floor dungeon should execute. Verify all parts connect: rooms generate, combat works, boss appears after floor 3, summary prints.
