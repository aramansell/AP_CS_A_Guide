Student RPG Guide: Dungeon Loop System

This is YOUR guide. After studying while-loops and for-loops, you wrap your combat system in loops so fights last multiple rounds and dungeons have multiple rooms. Instead of code blanks, you are provided with Acceptance Criteria: if your code passes every test, it works.

What You're Building

A DungeonLoop.java class that demonstrates two kinds of loops: a while loop for combat (keep fighting until someone dies) and a for loop for room exploration (five rooms, each with a random encounter). By the end, you understand when to use while vs for.

How It Connects to the Unit 4 Concepts

while-loops: Combat is indeterminate — you don't know how many rounds it will take. That's a while loop: "while both are alive, keep fighting."
for-loops: Dungeon rooms are fixed — you know exactly how many rooms to generate. That's a for loop: "for room 1 through 5, generate an encounter."

FILE: DungeonLoop.java

Create a new file called DungeonLoop.java. Every part on this page belongs to this ONE file.

Part 1: runCombat() — Combat While Loop

Recall while-loops and Combat.java (Unit 3). A while loop runs as long as its condition is true. Combat uses while (playerHP > 0 && monsterHP > 0) — the && operator from Unit 3's boolean logic.

Requirements:

  • Method signature: 
    public static boolean runCombat(String playerName, int playerHP, int playerAC,
                                    int playerAttackBonus, String monsterName,
                                    int monsterHP, int monsterAC,
                                    int monsterAttackBonus, int monsterDamageDie)
  • Prints a header: "=== COMBAT: [playerName] vs [monsterName] ==="
  • Uses a while loop. Condition: both playerHP > 0 AND monsterHP > 0
  • Each iteration:
    1. Print current HP for both combatants
    2. Player attacks: roll d20, add bonus, compare to AC. Same pattern from Combat.java Unit 3: if roll == 20, critical hit (roll d8 × 2). If total >= AC, hit (roll d8). Otherwise, miss.
    3. Subtract damage from monsterHP if hit
    4. Monster counterattacks ONLY if monsterHP > 0
    5. Monster attack: roll d20, add bonus, compare to AC. If total >= playerAC, hit (roll monsterDamageDie). Otherwise, miss.
    6. Subtract damage from playerHP if hit
  • After the loop ends (someone died): if playerHP > 0, print victory message and return true. Otherwise, print defeat message and return false

Socratic Checkpoint:

Why does the monster only counterattack when monsterHP > 0? What would happen if you removed that check?

Acceptance Test:

boolean result = DungeonLoop.runCombat("Hero", 20, 14, 3, "Goblin", 10, 12, 2, 6);
System.out.println("Combat result: " + (result ? "Won" : "Lost"));
// Run multiple times. Verify:
// - Combat continues until someone's HP reaches 0
// - Winner is reported correctly
// - The returned boolean matches the winner

When to Use While

You don't know how many rounds this fight will last. It could be 1 round or 20. That's exactly when you use a while loop — the number of iterations is determined by the data, not by a fixed count.

Part 2: runDungeon() — Room Exploration For Loop

Recall for-loops. When you know the count in advance, use for (int i = start; i <= end; i++). Five dungeon rooms = five iterations.

Requirements:

  • Method signature: public static void runDungeon(int difficulty)
  • Prints a header: "=== DUNGEON CRAWL (Difficulty: [difficulty]) ==="
  • Track playerHP (start at 20), gold (start at 0), roomsSurvived (start at 0)
  • Use a for loop: room from 1 to 5 (inclusive)
  • At the top of each iteration: if playerHP <= 0, print "You cannot continue..." and break
  • Each iteration generates a random room type (0, 1, or 2):
    • Combat (type 0): Scale a monster. monsterHP = 5 + room * difficulty, AC = 10 + room, atk = room + difficulty, dmg die = 4 + room. Print "A [name] appears!" (use an array of names indexed by room-1). Call runCombat(). If the player wins, add 10 * room gold. If not, set playerHP = 0.
    • Treasure (type 1): Add 5 + Dice.roll(10) * difficulty gold. Print what was found.
    • Trap (type 2): Subtract Dice.roll(6) from playerHP. Print the damage and remaining HP.
  • Increment roomsSurvived after processing each room
  • After the loop: print a summary showing rooms explored, gold collected, and HP remaining

Socratic Checkpoint:

Why is the dungeon loop a for loop but the combat loop is a while loop? What's the rule for choosing between them?

Acceptance Test:

DungeonLoop.runDungeon(1);
// Verify:
// - Exactly 5 rooms are generated (or fewer if player dies)
// - Room types vary randomly (re-run to see all types)
// - Combat, treasure, and trap rooms behave as described
// - HP carries over across rooms
// - Gold accumulates across rooms
// - If player dies, remaining rooms are skipped (break)

Part 3: main() — Test It

Requirements:

  • First, call runCombat() once with hardcoded stats to test while loops in isolation
  • Then, call runDungeon(1) to test the full for-loop dungeon

Acceptance Test:

Run DungeonLoop.main(). Confirm the isolated combat test runs first, then the full dungeon. Re-run — results should vary because dice rolls are random.
